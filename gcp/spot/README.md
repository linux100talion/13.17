# gcp/spot — ловля SPOT T4 по зонам + менеджмент дисков/снапшотов

Набор скриптов для работы с **вытесняемым (SPOT)** GPU-инстансом
`dev-workspace-1317` (Tesla T4) в регионе `europe-west4`. SPOT дешевле и чаще
доступен, но Google может его вытеснить в любой момент. Главная сложность —
**boot-диск зонален**, и вокруг этого крутится вся логика переноса состояния
между зонами через снапшоты. Этот README — про управление дисками и снапшотами;
параметры инстанса и общие конвенции — в `../CLAUDE.md`.

## Скрипты

По каждой из трёх зон региона (`a`, `b`, `c` — других в `europe-west4` нет) —
тройка скриптов, отличающихся внутри зоны только `TARGET_ZONE`:

| Скрипт | Что делает |
|---|---|
| `{a,b,c}_start.sh` | поймать/поднять SPOT T4 в своей зоне (с переносом диска, если нужно) |
| `{a,b,c}_stop.sh`  | погасить инстанс в своей зоне (диск остаётся, платишь только за storage) |
| `{a,b,c}_ssh.sh`   | SSH в инстанс своей зоны (доп. аргументы → `gcloud compute ssh`) |

Соседние скрипты верхнего уровня (`../01_create_workspace.sh`,
`../02_power_manager.sh`, `../04_check_money_leak.sh`) зовутся через `../`.

Тогглы `*_start.sh` (env): `SPOT=0` (on-demand вместо spot), `MACHINE_TYPE=…`
(меньше host'ов → реже EXHAUSTED), `REFRESH=1` (перезалить устаревший диск
целевой зоны из свежего снапшота).

## Почему всё крутится вокруг дисков

**Boot-диск зонален.** Persistent-диск GCE живёт в конкретной зоне и прицепить
его к инстансу в ДРУГОЙ зоне нельзя. SPOT-ёмкость T4 плавает по зонам: сегодня
свободно в `a`, завтра только в `c`. Значит, чтобы поймать спот в зоне, где
сейчас есть ёмкость, нужно сначала перенести туда boot-диск (со всей репой,
образами и работой). Прямого «move disk to another zone» в GCE нет — перенос
делается **через снапшот**:

```
свежий диск (зона X) ──snapshot──► снапшот (глобальный)
                                        │
                                        └─create──► новый диск (зона Y)
```

Снапшоты **глобальны** (не привязаны к зоне) — поэтому служат мостом между
зонами. Из одного снапшота можно создать диск в **любой зоне любого региона**
проекта.

Нюансы:

- **Storage location снапшота** (где физически лежат блоки — зональный /
  региональный / мультирегиональный) влияет на цену хранения и скорость
  восстановления (создание диска в «дальней» зоне медленнее тянет данные), но
  **не ограничивает**, в какой зоне можно создать диск — создать можно везде.
- При создании диска из снапшота **тип и размер можно поменять** (скрипт
  ставит `--type=pd-balanced`; размер — только в большую сторону).
- Снапшоты **инкрементальные**: первый полный, дальше только дельта — поэтому
  рабочий `dev-workspace-1317-snap` дёшево пересоздавать при каждом переезде.
- Зональность boot-диска обходится именно потому, что снапшот глобальный.
  Перенос между **проектами** — отдельная история (нужны права/шаринг), но в
  пределах нашего проекта зона значения не имеет.

## Что такое «свежий» диск (канон состояния)

В разных зонах может оказаться **несколько** дисков с именем `dev-workspace-1317`
(остатки прошлых переездов). Канон — диск инстанса, **работавшего последним**:
именно на нём последняя версия репы/образов/работы. Остальные зональные диски —
потенциально устаревшие снимки прошлого.

`*_start.sh` определяет источник состояния так:
1. если инстанс существует (в любой зоне) → его boot-диск и есть свежий;
2. иначе берётся диск с этим именем, предпочитая `FALLBACK_ZONE` (`europe-west4-a`,
   где исторически лежит исходный диск);
3. если ни инстанса, ни диска нет — грузиться не с чего, нужен
   `../01_create_workspace.sh`.

## Логика `*_start.sh` по шагам

1. **Инстанс уже в целевой зоне?**
   - `RUNNING` → ловить нечего, выходим.
   - остановлен → `instances start` (быстрый возврат из preemption, без снапшота).
   - `EXHAUSTED` на старте → в зоне нет свободных T4, пробуем соседнюю зону.
2. **Определить свежий источник** (см. выше).
3. **Обеспечить boot-диск в целевой зоне:**
   - свежий диск уже в целевой зоне → снапшот не нужен;
   - в целевой зоне есть диск, но свежий — в другой → СТОП с просьбой `REFRESH=1`
     (чтобы случайно не загрузиться с устаревшего диска);
   - иначе: снапшот свежего → создать диск в целевой зоне из снапшота
     (при `REFRESH=1` сначала удаляется устаревший диск целевой зоны).
4. **Освободить имя инстанса** в другой зоне: `delete --keep-disks=boot`
   (имя `dev-workspace-1317` уникально в проекте; диск той зоны сохраняется).
5. **Создать SPOT T4-инстанс** в целевой зоне на подготовленном диске; при
   `RESOURCE_POOL_EXHAUSTED` инстанс не создан, но диск целевой зоны готов —
   повторить позже эту же зону или пробовать соседнюю.

Рабочий снапшот переноса — `dev-workspace-1317-snap`, пересоздаётся при каждом
переезде (старый удаляется перед созданием нового).

## ⚠️ Уборка дисков — на тебе

Скрипты **НЕ удаляют** старые зональные диски сами (исключение — `REFRESH=1`,
который убивает устаревший диск целевой зоны перед перезаливкой). После
нескольких переездов в зонах копятся диски `dev-workspace-1317`, **за каждый
идёт плата за storage** (даже если инстанс выключен). Канон — только диск
последней рабочей зоны; остальные чистить вручную:

```bash
# посмотреть все диски с этим именем по зонам:
gcloud compute disks list --filter="name=dev-workspace-1317" \
  --project=drone-13-17-workspace-2026

# удалить устаревший диск в конкретной зоне:
gcloud compute disks delete dev-workspace-1317 --zone=<zone> \
  --project=drone-13-17-workspace-2026
```

Снапшот `dev-workspace-1317-snap` тоже занимает место, но он переиспользуется
(пересоздаётся при каждом переезде), так что отдельной уборки обычно не требует.

## Типовые сценарии

**Поймать спот (ёмкость плавает по зонам):** гоняй по очереди до успеха —
```bash
./a_start.sh || ./b_start.sh || ./c_start.sh
```

**Вытеснило (termination=STOP) → вернуться:** инстанс остановлен в своей зоне,
диск там же — просто подними его снова тем же скриптом зоны (быстрый `start` без
снапшота): `./a_start.sh`.

**Закончил работу:** `./a_stop.sh` (диск цел, платишь только за storage).

**Подключиться:** `./a_ssh.sh` или с командой `./a_ssh.sh --command='nvidia-smi'`.

**Переехать в другую зону (в текущей нет T4):** запусти `*_start.sh` целевой
зоны — он сам снимет снапшот свежего диска, создаст диск в целевой зоне и
освободит имя инстанса. После переезда **не забудь удалить** оставшийся диск
старой зоны (плата за storage).
