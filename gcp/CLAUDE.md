# gcp/ — управление dev-инстансом в Google Compute Engine

Набор bash-обёрток над `gcloud` для одной GPU-машины (Tesla T4) на GCE.
Цель — поднимать сервер только под работу и подгонять мощность под задачу
ради экономии бюджета.

## Параметры инстанса (захардкожены во всех скриптах)

| Параметр   | Значение                     |
|------------|------------------------------|
| Project    | `drone-13-17-workspace-2026` |
| Zone       | `europe-west4-a` (default, env-override `ZONE=…`) |
| Instance   | `dev-workspace-1317`          |
| GPU        | 1× NVIDIA Tesla T4           |
| OS         | Ubuntu 24.04 LTS (amd64)     |
| Boot disk  | 120 GB, pd-balanced          |

## Скрипты

- `01_create_workspace.sh [DISK_GB]` — первичное создание инстанса
  (n1-standard-8, T4, диск `DISK_GB` GB, по умолчанию 120). Запускается один раз.
  `--maintenance-policy=TERMINATE` обязателен для машин с GPU. Тоггл
  `GPU=0 ./01_create_workspace.sh` — **CPU-only build-box** (без `--accelerator`,
  доступен сразу, не упирается в дефицит T4; см. раздел «Build-box»).
- `02_power_manager.sh {start|stop|ssh}` — повседневное питание. Остановленный
  инстанс не тарифицируется за CPU/GPU (платишь только за диск).
- `03_resize_node.sh {light|heavy}` — смена типа машины «на лету»
  (стоп → set-machine-type → старт): `light` = n1-standard-4, `heavy` = n1-standard-8.
- `04_check_money_leak.sh` — статус инстанса (`RUNNING` / `TERMINATED`).
- `05_delete_instance.sh` — удаление с подтверждением `yes`; загрузочный диск
  сохраняется (`--keep-disks=boot`).
- `06_build_sizes.sh` — сколько места реально съела сборка: по SSH гоняет
  `docker system df`/`docker images` на инстансе (образы проекта, build-кэш,
  диск). Только чтение, требует RUNNING-инстанс.
- `07_list_instances.sh` — список всех инстансов проекта (имя/зона/тип/статус/
  spot/external-IP). Только чтение; пустой вывод = инстансов нет.

## Конвенции

- **Работаем только существующими скриптами и их комбинациями**, а не разовыми
  ручными `gcloud`-командами. Если нужного скрипта нет — сначала пишем его,
  коммитим, пушим и только потом пользуемся. Любое повторяемое действие
  остаётся в репе как скрипт.
- Все скрипты — тонкие обёртки над `gcloud`; общие константы
  (`PROJECT`/`INSTANCE_NAME`) дублируются в каждом файле — при правке менять во
  всех. `ZONE` переопределяется из окружения: `ZONE="${ZONE:-europe-west4-a}"`,
  поэтому зону для разовой пробы задаём не правкой файлов, а `ZONE=europe-west4-b
  ./01_create_workspace.sh` (удобно при `RESOURCE_POOL_EXHAUSTED` — перебрать
  зоны под T4).
- Деструктивные операции спрашивают подтверждение (только `05`). Остальные
  безопасны для повторного запуска.
- Требуется настроенный `gcloud` CLI и `gcloud auth login` с доступом к проекту.
- Креды (`*.json`, `*.pem`, `*.key`) в репозиторий не коммитим — см. `.gitignore`.

## Build-box: что можно и нельзя проверять, чтобы не привязаться

CPU-only инстанс (`GPU=0 ./01_create_workspace.sh`) — это **верстак для проверки
кода**, а не уменьшенная копия дрона. Три реальных таргета различаются по железу:
Orin Nano (ARM64, Ampere sm_87, JetPack/L4T), ноут (x86, Ada sm_89, RTX 4060),
GCP (x86). Чтобы build-box не привязал к своему окружению — держим границы:

**МОЖНО (и ради этого он и нужен):**
- компиляция C++/CUDA, `colcon build`, сборка docker-образов, линт, статанализ,
  синтаксис скриптов. `nvcc` собирает CUDA-код БЕЗ физического GPU.

**НЕЛЬЗЯ (нужно реальное железо целевого класса):**
- исполнять CUDA-кернелы, инференс torch/DINOv2, CUDA-дебайер на живом кадре,
  Gazebo с рендером. Зелёная сборка ≠ «поедет на Orin»: runtime-ошибки CUDA
  (illegal access, арх-интринсики) ловит только GPU нужной арх.

**Почему привязка тут МАЛОВЕРОЯТНА (и что её гарантирует):**
- нет GPU → нет cmake-автодетекта «native arch» → нельзя втихую зашить одну арх;
- сборка идёт ВНУТРИ контейнера → CUDA/драйвер/OpenGL **хоста** в неё не текут;
  версия CUDA = тег базового образа (один и тот же на ноуте/GCP), запинен 12.2.2,
  что совпадает с JetPack 6 / L4T r36 (Orin);
- OpenGL для проверки кода вообще ни при чём (он только для рендера Gazebo).

**Дисциплина, которую держать:**
- базовые образы — **по запиненному тегу**, не `:latest`;
- `CUDA_ARCH_BIN` — **явный multi-arch** build-arg, никогда `native`/autodetect;
- Orin собирается своим путём (`docker/orin`, база L4T, sm_87) — x86-образ
  `docker/sim/nav` про Orin знать не обязан, это и есть страховка от арх-привязки.
