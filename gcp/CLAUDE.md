# gcp/ — управление dev-инстансом в Google Compute Engine

Набор bash-обёрток над `gcloud` для одной GPU-машины (Tesla T4) на GCE.
Цель — поднимать сервер только под работу и подгонять мощность под задачу
ради экономии бюджета.

## Параметры инстанса (захардкожены во всех скриптах)

| Параметр   | Значение                     |
|------------|------------------------------|
| Project    | `drone-13-17-workspace-2026` |
| Zone       | `europe-west4-a` (default, env-override `ZONE=…`) |
| Instance   | `dev-workspace-1317`          |
| GPU        | 1× NVIDIA Tesla T4           |
| OS         | Ubuntu 24.04 LTS (amd64)     |
| Boot disk  | 120 GB, pd-balanced          |

## Скрипты

- `01_create_workspace.sh [DISK_GB]` — первичное создание инстанса
  (n1-standard-8, T4, диск `DISK_GB` GB, по умолчанию 120). Запускается один раз.
  `--maintenance-policy=TERMINATE` обязателен для машин с GPU. Тоггл
  `GPU=0 ./01_create_workspace.sh` — **CPU-only build-box** (без `--accelerator`,
  доступен сразу, не упирается в дефицит T4; см. раздел «Build-box»).
- `02_power_manager.sh {start|stop|ssh}` — повседневное питание. Остановленный
  инстанс не тарифицируется за CPU/GPU (платишь только за диск).
- `03_resize_node.sh {light|heavy}` — смена типа машины «на лету»
  (стоп → set-machine-type → старт): `light` = n1-standard-4, `heavy` = n1-standard-8.
- `04_check_money_leak.sh` — статус инстанса (`RUNNING` / `TERMINATED`).
- `05_delete_instance.sh` — удаление с подтверждением `yes`; загрузочный диск
  сохраняется (`--keep-disks=boot`).
- `06_build_sizes.sh` — сколько места реально съела сборка: по SSH гоняет
  `docker system df`/`docker images` на инстансе (образы проекта, build-кэш,
  диск). Только чтение, требует RUNNING-инстанс.
- `07_list_instances.sh` — список всех инстансов проекта (имя/зона/тип/статус/
  spot/external-IP). Только чтение; пустой вывод = инстансов нет.
- `08_add_gpu.sh` — апгрейд CPU-only build-box до GPU-машины **пересозданием
  с сохранением boot-диска**. Прикрепить GPU к существующему инстансу gcloud не
  умеет (subcommand `set-accelerator` не существует; REST `instances.update`
  с `guestAccelerators` отдаёт 503), поэтому единственный надёжный путь —
  стоп → `delete --keep-disks=boot` → `create` с тем же диском + `--accelerator`
  + `TERMINATE`. Весь диск (репа, образы, работа) переживает пересоздание.
  Идемпотентен: если инстанса уже нет (прошлый прогон упал после удаления),
  просто пересоздаёт из сохранённого диска. После старта печатает команды
  установки `nvidia-driver-535` + `nvidia-container-toolkit` внутри инстанса.
  Тогглы против дефицита T4 (диск НЕ переносят, остаётся в той же зоне):
  `MACHINE_TYPE=n1-standard-4 ./08_add_gpu.sh` (меньше host'ов → реже EXHAUSTED),
  `SPOT=1 ./08_add_gpu.sh` (Spot-пул: чаще доступен и дешевле, но вытесняемый;
  termination=STOP → при вытеснении инстанс останавливается, диск цел),
  комбинируются. При RESOURCE_POOL_EXHAUSTED инстанс не создаётся, но boot-диск
  СОХРАНЁН — повторить позже `./08_add_gpu.sh`. Диск привязан к зоне: сменить
  зону можно только снапшотом диска (отдельная история).
- `09_quota_manager.sh {show|ensure [VALUE]}` — просмотр и запрос GPU-квот.
  Для создания GPU-VM должны быть ненулевыми СРАЗУ несколько квот: глобальная
  `GPUS_ALL_REGIONS` (часто = 0 у новых проектов и режет всё, даже при наличии
  региональной) + региональные `NVIDIA_T4_GPUS` (on-demand) и
  `PREEMPTIBLE_NVIDIA_T4_GPUS` (spot). `show` печатает лимиты/usage; `ensure
  [VALUE]` (default 1) запрашивает повышение через Cloud Quotas API
  (`gcloud alpha quotas preferences create`) ТОЛЬКО где лимит ниже нужного
  (идемпотентно). Запрос идёт на ревью Google — для 1×T4 на проекте с биллингом
  обычно аппрувится за секунды. Регион берётся из `ZONE` (env-override).
- **`spot/`** — «ловля» SPOT T4-инстанса по трём зонам региона europe-west4
  (a/b/c; других зон в регионе нет). По каждой зоне — тройка скриптов
  `{a,b,c}_{start,stop,ssh}.sh` (внутри зоны отличаются только `TARGET_ZONE`).
  Соседние скрипты верхнего уровня зовутся как `../01_…`, `../02_…`, `../04_…`.
  - `{a,b,c}_start.sh` — поймать спот в своей зоне. Стратегия: гонять по очереди
    a → b → c, пока зона не отдаст T4. boot-диск зонален — поймать спот в чужой
    зоне можно только перенеся диск **снапшотом**, что скрипт делает сам: ① если
    инстанс уже в целевой зоне — `start` (быстрый возврат из preemption);
    ② определяет самый свежий источник (диск работавшего последним инстанса,
    иначе диск в `a`); ③ обеспечивает boot-диск в целевой зоне (свежий уже там →
    без снапшота; потенциально устаревший диск в зоне → стоп с просьбой
    `REFRESH=1`; иначе снапшот свежего → зональный диск); ④ освобождает имя
    инстанса в другой зоне (`delete --keep-disks=boot`); ⑤ создаёт SPOT
    T4-инстанс, ловит `RESOURCE_POOL_EXHAUSTED`. Тогглы: `SPOT=0` (on-demand),
    `MACHINE_TYPE=…`, `REFRESH=1` (перезалить устаревший диск целевой зоны).
    ⚠️ Старые зональные диски сам НЕ удаляет (кроме `REFRESH=1`) — за них идёт
    плата за storage, чистить вручную `gcloud compute disks delete`.
  - `{a,b,c}_stop.sh` — погасить инстанс в своей зоне (boot-диск остаётся там же,
    платишь только за storage). Пропускает уже `TERMINATED`; если инстанс в
    другой зоне — подсказывает нужный скрипт. Парный к `*_start.sh`.
  - `{a,b,c}_ssh.sh` — SSH в инстанс своей зоны (требует `RUNNING`); доп.
    аргументы пробрасываются в `gcloud compute ssh` (напр. `./a_ssh.sh
    --command='nvidia-smi'`). Тонкая обёртка над `../02_power_manager.sh ssh`
    с зоной, прибитой к a/b/c.
  - `{a,b,c}_convert.sh` — перевести существующий on-demand-инстанс в SPOT
    **in-place** через `set-scheduling` (стоп при необходимости →
    `--provisioning-model=SPOT --instance-termination-action=STOP`, без
    пересоздания). Зеркально к `on_demand/{a,b,c}_convert.sh` (обратное
    направление).
- **`on_demand/`** — питание **on-demand** (не вытесняемого) T4-инстанса по тем
  же трём зонам: `{a,b,c}_{start,stop,ssh}.sh` (отличаются только `TARGET_ZONE`).
  Зеркало `spot/` по раскладке, но скрипты **тонкие** — никакой снапшот-логики и
  переезда между зонами (это забота `spot/`). `start` = только `gcloud instances
  start` существующего инстанса в своей зоне (создание — `01_…`/`08_…`, перенос
  диска — `spot/`); `stop`/`ssh` — как у `spot/`. `{a,b,c}_convert.sh` — перевести
  существующий SPOT-инстанс в on-demand (STANDARD) **in-place** через
  `set-scheduling` (стоп при необходимости → `--provisioning-model=STANDARD
  --no-preemptible`, диск/инстанс на месте; альтернатива пересозданию через
  `08_add_gpu.sh`). Когда что: on-demand живёт постоянно и не вытесняется
  (дороже) → `on_demand/`; ловля дешёвой вытесняемой ёмкости с переездом диска по
  зонам → `spot/`. Подробности — `on_demand/README.md`.
- **`snapshot/`** — работа с бэкап-снапшотами boot-диска по зонам.
  `{a,b,c}_take.sh` — снять датированный **бэкап**-снапшот в своей зоне
  (`<instance>-snap-YYYYMMDD`, без буквы зоны — диск живёт в одной зоне за раз,
  снапшот глобальный; storage location региональный, инкрементальный,
  идемпотентно). Это НЕ транзитный `<instance>-snap` из `spot/`
  — независимые долгоживущие снимки; существующие не трогаются. `{a,b,c}_restore.sh`
  — восстановить диск в своей зоне из снапшота (по умолчанию самый свежий,
  `SNAPSHOT=…` override); существующий диск молча не затирает: занятый инстансом
  → отказ, свободный → только `REPLACE=1`. Снапшот глобален → восстановить можно
  в любой зоне. `list.sh` (ОДИН, не a/b/c — снапшоты глобальны) — снапшоты диска
  + сводка дисков по зонам, только чтение. Уборку старых снапшотов скрипты не
  делают — вручную. Подробности — `snapshot/README.md`.

## Конвенции

- **Работаем только существующими скриптами и их комбинациями**, а не разовыми
  ручными `gcloud`-командами. Если нужного скрипта нет — сначала пишем его,
  коммитим, пушим и только потом пользуемся. Любое повторяемое действие
  остаётся в репе как скрипт.
- Все скрипты — тонкие обёртки над `gcloud`; общие константы
  (`PROJECT`/`INSTANCE_NAME`) дублируются в каждом файле — при правке менять во
  всех. `ZONE` переопределяется из окружения: `ZONE="${ZONE:-europe-west4-a}"`,
  поэтому зону для разовой пробы задаём не правкой файлов, а `ZONE=europe-west4-b
  ./01_create_workspace.sh` (удобно при `RESOURCE_POOL_EXHAUSTED` — перебрать
  зоны под T4).
- Деструктивные операции спрашивают подтверждение (только `05`). Остальные
  безопасны для повторного запуска.
- Требуется настроенный `gcloud` CLI и `gcloud auth login` с доступом к проекту.
- Креды (`*.json`, `*.pem`, `*.key`) в репозиторий не коммитим — см. `.gitignore`.

## Build-box: что можно и нельзя проверять, чтобы не привязаться

CPU-only инстанс (`GPU=0 ./01_create_workspace.sh`) — это **верстак для проверки
кода**, а не уменьшенная копия дрона. Три реальных таргета различаются по железу:
Orin Nano (ARM64, Ampere sm_87, JetPack/L4T), ноут (x86, Ada sm_89, RTX 4060),
GCP (x86). Чтобы build-box не привязал к своему окружению — держим границы:

**МОЖНО (и ради этого он и нужен):**
- компиляция C++/CUDA, `colcon build`, сборка docker-образов, линт, статанализ,
  синтаксис скриптов. `nvcc` собирает CUDA-код БЕЗ физического GPU.

**НЕЛЬЗЯ (нужно реальное железо целевого класса):**
- исполнять CUDA-кернелы, инференс torch/DINOv2, CUDA-дебайер на живом кадре,
  Gazebo с рендером. Зелёная сборка ≠ «поедет на Orin»: runtime-ошибки CUDA
  (illegal access, арх-интринсики) ловит только GPU нужной арх.

**Почему привязка тут МАЛОВЕРОЯТНА (и что её гарантирует):**
- нет GPU → нет cmake-автодетекта «native arch» → нельзя втихую зашить одну арх;
- сборка идёт ВНУТРИ контейнера → CUDA/драйвер/OpenGL **хоста** в неё не текут;
  версия CUDA = тег базового образа (один и тот же на ноуте/GCP), запинен 12.2.2,
  что совпадает с JetPack 6 / L4T r36 (Orin);
- OpenGL для проверки кода вообще ни при чём (он только для рендера Gazebo).

**Дисциплина, которую держать:**
- базовые образы — **по запиненному тегу**, не `:latest`;
- `CUDA_ARCH_BIN` — **явный multi-arch** build-arg, никогда `native`/autodetect;
- Orin собирается своим путём (`docker/orin`, база L4T, sm_87) — x86-образ
  `docker/sim/nav` про Orin знать не обязан, это и есть страховка от арх-привязки.
