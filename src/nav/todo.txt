ЗАДАЧА
------
Взлетать в ALT_HOLD и инициализировать VINS в полёте (реальным движением), без
GPS. Цель этапа — посмотреть UX (как выглядит сама инициализация + позже рывок на
handover), потом решать, что делать с расхождением нуля VINS и датума.

ПОЧЕМУ ALT_HOLD, А НЕ GUIDED
----------------------------
GUIDED — позиционный режим: требует валидную горизонтальную позицию EKF, которой
без GPS и без сошедшегося VINS нет → mode даже не залатчится. ALT_HOLD держит
высоту по барометру и НЕ требует горизонтальной позиции (только IMU+компас+баро).
Курица-яйцо монокуляра: VINS нужно движение для init, а движение даёт сам взлёт.

НЮАНС ПО ГАЗУ (ALT_HOLD, стандартное поведение)
-----------------------------------------------
Стик газа ПРУЖИННЫЙ, возвращается в ЦЕНТР. Газ НЕ нужно удерживать:
  - газ вверх  → дрон поднимается;
  - газ отпустили → стик в центре → дрон ДЕРЖИТ высоту.
В терминах RC override: throttle=1650 (climb) → вернуть throttle=1500 (центр=hold).
Центр (1500) и есть «держать высоту» — отдельный «hover PWM» подбирать НЕ нужно.

ДИЗАЙН (одобрено)
-----------------
Отличие от текущих команд: arm/takeoff/square работают в GUIDED (одноразовый
cmd/takeoff + setpoint_position, дрон сам держит точку). ALT_HOLD авто-взлёта не
имеет — высота управляется throttle-стиком → нужен НЕПРЕРЫВНЫЙ RC override
(/mavros/rc/override, ~20 Гц), которого в проекте ещё нет. Непрерывный паблиш с
меняющимися значениями плохо ложится на bash → делаем Python-нодой (по образцу
fly_square.py), bash — тонкая обёртка для секвенсора (как square.sh).

1) src/lab/alt_hold_bootstrap.py — нода-конечный автомат (use_sim_time):
     ALT_HOLD → arm (throttle=min) → CLIMB (throttle>center до целевой высоты)
       → HOLD+EXCITE (throttle=center=1500, импульсы roll/pitch для параллакса)
       → ждать сходимости VINS (подписка /vins_estimator/odometry, N сообщений
         подряд — НЕ grep лога)
       → [--handover] переключить в GUIDED → наблюдать UX (рывок); по умолчанию
         БЕЗ handover (сначала чистое наблюдение init), handover — отдельный прогон
   - высота: /mavros/global_position/rel_alt (баро, доступна БЕЗ origin/GPS, в
     отличие от local_position, которому нужна горизонтальная оценка);
   - сходимость VINS: устойчивый поток /vins_estimator/odometry;
   - бюджеты ожидания — в SIM-времени по /clock (как arm.sh/hover.sh), RTF-незав.;
   - excitation: одного вертикального climb монокуляру может не хватить → мягкие
     импульсы roll/pitch (±небольшой PWM от центра);
   - аргументы (argparse): --alt, --handover, --excite (амплитуда), --excite-period,
     --vins-timeout, --throttle-climb (1650), --throttle-hold (1500).

2) src/lab/bootstrap.sh — обёртка (docker exec … python3 …) для встраивания в
   capture_scene.sh как команда `bootstrap` между рестартом и square/land.

ФЛАГ --handover
   off (default): нода доводит до сходимости VINS и ОСТАНАВЛИВАЕТСЯ на HOLD,
                  логирует факт init (чистое наблюдение, без рывка).
   on:            после сходимости переключает в GUIDED — увидеть рывок/расхождение.

РИСКИ / ПРОВЕРИТЬ НА ПРОГОНЕ
---------------------------
- RC override в этом SITL может требовать совпадения SYSID_MYGCS либо быть прикрыт
  → заложить диагностику (читать обратно /mavros/rc/in) и явный лог, если FCU не
  принимает override.
- Хватит ли вертикального climb + лёгкой раскачки для init монокуляра (иначе
  увеличить амплитуду/длительность excitation).

СТАТУС РЕАЛИЗАЦИИ (ветка nn2_c3_vins_althold)
---------------------------------------------
СДЕЛАНО:
- src/lab/alt_hold_bootstrap.py — нода-конечный автомат (use_sim_time, 20 Гц RC
  override): PREARM(ALT_HOLD,газ=min) → ARM → CLIMB(газ>центр до --alt) →
  EXCITE(газ=центр + импульсы roll/pitch) → ждёт сходимости VINS → по --handover:
  GUIDED (наблюдать рывок) либо OBSERVE→LAND (самодостаточно, без рывка). Дефолт —
  без handover.
- src/lab/bootstrap.sh — обёртка с BS_* env (по образцу square.sh).
- capture_scene.sh — `bootstrap` в валидации + проброс BS_* env.
- docker/sim/Makefile — цель `make bootstrap` (+ .PHONY).
- src/lab/CLAUDE.md — секция с таблицей env и предупреждениями.

КЛЮЧЕВЫЕ РЕШЕНИЯ:
- ALT_HOLD требует НЕПРЕРЫВНОГО override → нода владеет всем полётом, arm/takeoff
  перед ней не нужны; завершает ALT_HOLD-фазу только переходом в самоподдерживающийся
  режим (GUIDED/LAND) — «зависнуть и выйти» небезопасно.
- Газ: climb=1650 → hold=ЦЕНТР 1500 (пружинный стик, отдельный «hover PWM» не нужен).
- Высота по rel_alt (баро, без origin), сходимость VINS по потоку /odometry, бюджеты
  в sim-времени.

ЗАПУСК (после подъёма стека):
  make bootstrap                          # climb→init→observe→land, без рывка
  make bootstrap BS_ALT=4 BS_HANDOVER=1   # после init → GUIDED (смотреть рывок)

ПРОВЕРИТЬ НА 1-м ПРОГОНЕ (диагностика заложена):
  1. Принимает ли SITL RC override — нода логит `rc/in throttle=…` в CLIMB; если
     высота не растёт → override не проходит (возможно нужен SYSID_MYGCS либо флип
     RC_NOCHANGE=65535 в ноде). Аборт→LAND отработает безопасно.
  2. Хватает ли climb+раскачки монокуляру для init — иначе поднять BS_EXCITE /
     --excite-period.

СЛЕДУЮЩЕЕ ЛОГИЧНОЕ:
