# FAQ — выключение GPS, External Nav и origin (GPS-denied арминг/взлёт)

Конспект обсуждения: как правильно «выключить GPS» в проекте, чтобы арминг не ждал
позиционную оценку, а GUIDED-взлёт/квадрат работали на внешней навигации
(`vision_pose` от `ray_tracer`). Совпадает с боевой архитектурой проекта
(ray_tracer → `/mavros/vision_pose/pose`, на FCU `EK3_SRC1_POSXY=6`).

---

## Проблема

Арминг происходит долго — стек ждёт позиционную оценку (GPS-лок). Нужно обойти GPS,
чтобы арм не ждал. Но чтобы это работало и для последующих GUIDED-взлёта/квадрата
(а не только для самого факта арма), правильный путь — дать EKF позицию из внешнего
источника (`vision_pose`) с момента старта, не дожидаясь лока GPS.

Дополнительно: у нас будут якорные точки с псевдо-GPS координатами, и точка взлёта
тоже является якорной точкой с псевдо-GPS координатами (заданными вручную). Вопрос:
можно и стоит ли задать начальные координаты GPS вручную?

---

## Что сейчас на самом деле происходит

В `docker/sim/config/sitl-extra.parm` **нет** ни одного параметра про GPS/EKF-источник.
То есть GPS по-прежнему включён (дефолт `EK3_SRC1_POSXY=3` = GPS), а `ARMING_CHECK 0`
просто **маскирует симптом**: арм проходит, но EKF всё равно не имеет валидной
горизонтальной позиции. Поэтому:

- `takeoff.sh` сам признаётся в комментарии: *«Высота может быть пустой первые
  секунды (EKF без origin)»* — это и есть корень.
- `fly_square.py` (`setpoint_position/local`) без origin и без позиции работает
  плохо/непредсказуемо.

Диагноз: обходить надо не арм-чек, а **дать EKF позицию из внешнего источника с
момента старта**. `ray_tracer` для этого уже готов — он публикует `vision_pose`
непрерывно с каждой одометрии VINS (`offset=0` до первой засечки → сырой VINS), это
и есть нужный поток ExternalNav. Не хватает двух вещей: переключить источник EKF и
**задать датум**.

---

## Правильный путь = три вещи вместе

### 1. Выключить GPS и переключить EKF на ExternalNav (в `sitl-extra.parm`)

```
AHRS_EKF_TYPE  3      # EKF3
GPS_TYPE       0      # приёмник GPS выключен полностью
EK3_SRC1_POSXY 6      # позиция XY ← ExternalNav (vision_pose)
EK3_SRC1_VELXY 0      # vision_pose не несёт скорость → None (EKF выведет сам)
EK3_SRC1_POSZ  1      # высота ← Baro (надёжнее, чем extnav-Z)
EK3_SRC1_VELZ  0
EK3_SRC1_YAW   1      # курс ← компас (в sim есть; на борту позже vision-yaw=6)
EK2_ENABLE     0
```

Важный нюанс: `VELXY=0`, потому что `ray_tracer` шлёт `PoseStamped` без скорости.
Если поставить `6`, EKF будет ждать `vision_speed` и ругаться. (Позже, если захочешь,
VINS даёт `twist` → можно поднять отдельный `/mavros/vision_speed/speed_twist`.)

### 2. Поток `vision_pose` уже идёт с буста

Этот пункт закрыт (`ray_tracer`).

### 3. Задать origin EKF вручную

Вот это сейчас отсутствует — и это ответ на вопрос про ручные координаты.

---

## Да, начальные координаты надо задать вручную — это не костыль

Без GPS у EKF **нет абсолютного датума**, а ExternalNav даёт только *относительную*
позу (VINS стартует в 0,0,0). Пока origin не задан — EKF не выдаёт `local_position`,
`set_gp_origin` обязателен. Механизм в ArduPilot/MAVROS штатный:

```bash
ros2 topic pub --once /mavros/global_position/set_gp_origin \
  geographic_msgs/msg/GeoPointStamped \
  '{position: {latitude: <LAT>, longitude: <LON>, altitude: <ALT>}}'
```

Ключевой архитектурный момент (отвечает на «а стоит ли»):

**Точка взлёта — это якорь №0, и его псевдо-GPS координаты задаются вручную по
определению.** Их физически нельзя получить засечкой NN1 (ты ещё не взлетел, чтобы
увидеть ориентиры). Это датум всего проекта. Причём он **уже есть** — `ray_tracer`
читает `database.json → origin` (датум взлёта), относительно него
`geo.geodetic_to_enu` считает позиции всех якорей.

Значит правило простое и обязательное: **origin EKF на полётнике =
`database.json.origin`**, один источник истины. Тогда:

- VINS (0,0,0 в точке взлёта) ↔ EKF local NED ↔ псевдо-GPS-датум якорей — **один кадр**;
- засечка NN1 по якорю даёт ENU относительно того же датума → коррекция `vision_pose`
  логически непротиворечива;
- `global_position` дрона = origin + относительная поза — в той же псевдо-GPS системе,
  что и якорная база.

Если задать разные датумы для EKF и для базы якорей — кадры разъедутся, и первая же
засечка дёрнет дрон.

---

## Курс (yaw) — единственное, о чём ещё стоит подумать

Origin задаёт lat/lon/alt, но **ориентацию локального NED относительно мира задаёт
начальный yaw**. В симе есть симкомпас → `EK3_SRC1_YAW=1` решает это бесплатно. На
боевом борту компасу доверия мало → план `=6` (yaw из VINS/vision). Для текущих симов
оставляй компас, это развязывает узел.

---

## Порядок (важна последовательность)

В секвенсоре (`capture_scene.sh`/`arm.sh`) до арма:

```
1. set_gp_origin  ← LAT/LON/ALT из database.json.origin
2. дождаться, что /mavros/local_position/pose начал публиковаться
   (origin принят, vision_pose усвоен)
3. GUIDED → arm → takeoff → fly_square
```

После этого арм не ждёт GPS (его нет), а позиция есть сразу → можно даже вернуть
`ARMING_CHECK` к осмысленному значению, чтобы пайплайн валидировался, а не маскировался.

---

## Уменьшит ли задание origin время арминга?

Да, но точно разделим, что именно ускоряется.

**Что замедляет арм сейчас:** EKF без валидной горизонтальной позиции. В норме (с GPS)
арм ждёт GPS-лок + сходимость EKF — это и есть «долгие» секунды. С `ARMING_CHECK 0`
этот *чек* выключен, поэтому арм формально проходит. Но GUIDED-взлёт всё равно
утыкается в отсутствие позиции/origin.

**Что даёт origin:** сам по себе `set_gp_origin` — не «ускоритель арма» в чистом виде.
Он включает звено: как только задан датум **и** идёт `vision_pose`, EKF получает
горизонтальную позицию → начинает публиковать `local_position` → становится «healthy».
Это нужно, чтобы арм-чек позиции проходил быстро (если вернуть `ARMING_CHECK`), а
главное — чтобы взлёт/квадрат вообще работали.

| Сценарий | Время арма | Взлёт работает |
|---|---|---|
| Сейчас: `ARMING_CHECK 0`, GPS-дефолт, без origin | арм быстрый (чек выключен), но… | нестабильно (нет позиции) |
| GPS off + ExternalNav + origin + vision_pose | арм быстрый **по-честному** | да |

**Вывод:** уменьшение времени арма даёт **связка** (GPS off → EKF не ждёт лока,
которого не будет; origin + vision_pose → позиция появляется почти мгновенно). Origin —
необходимое, но не самостоятельное звено: без потока `vision_pose` он один позицию не
даст, и наоборот.

Тонкий момент по последовательности: задержка теперь переезжает на шаг «дождаться,
пока EKF усвоит origin + первый `vision_pose`» (пункт 2). Но это секунды и
детерминированно (поллинг `local_position`), а не неопределённое ожидание GPS-лока,
которого в GPS-denied не случится никогда. Так что суммарно — быстрее и предсказуемее.

---

## В каком режиме мы сейчас летаем

Весь цикл идёт в **GUIDED**, кроме посадки:

| Шаг | Режим | Чем делается |
|---|---|---|
| `arm` | **GUIDED** | `set_mode GUIDED` → `cmd/arming` |
| `takeoff` | **GUIDED** | `cmd/takeoff` (CommandTOL) |
| `hover` | **GUIDED** | ничего не шлём — ArduPilot сам удерживает точку (не PX4 OFFBOARD, continuous setpoints не нужны) |
| `fly_square` | **GUIDED** | `setpoint_position/local` @10 Гц |
| `land` | **LAND** | `set_mode LAND` |

Единственное переключение — в самом конце на LAND. Всё активное пилотирование — GUIDED.

Это важно для origin/ExternalNav: **GUIDED жёстко требует валидную позицию EKF** (и
для `cmd/takeoff`, и тем более для `setpoint_position/local`). Поэтому без заданного
датума + потока `vision_pose` GUIDED и «плывёт» — это ровно тот режим, который сильнее
всего страдает от отсутствия origin. С внешней навигацией он же будет работать штатно.

## Сейчас мы в GUIDED с GPS, но без VPS?

Почти — с одним уточнением. Пуск: `sim_nav.launch.py` включает `nav.launch.py`, а тот
поднимает `ray_tracer` → значит `vision_pose` в симе **публикуется**. Картина:

- **Режим:** GUIDED ✓
- **GPS:** да — симулированный GPS SITL включён по дефолту и **является текущим
  источником позиции EKF** (`EK3_SRC1_POSXY=3`, мы его не меняли).
- **VPS:** `ray_tracer` шлёт `/mavros/vision_pose/pose` (и MAVROS, скорее всего,
  форвардит на FCU), **но EKF его игнорирует** — источник стоит GPS, а не ExternalNav
  (`=6`).

Точная формулировка: **мы в GUIDED на симулированном GPS; `vision_pose` публикуется
вхолостую — EKF на него не смотрит.** То есть «без VPS» — не в смысле «не публикуется»,
а в смысле «не используется полётником».

Удобная стартовая точка: вся проводка VPS уже готова и крутится. Остаётся переключить
источник EKF (`POSXY=6` + `GPS_TYPE=0`) и задать origin — и тот же поток, что сейчас
идёт вхолостую, станет рабочим.

## В нашей модели дрона есть барометр?

Короткий ответ: **в Gazebo-модели барометра нет, но у полётника он есть.** Два уровня:

**1. Gazebo-модель (`worlds/iris_cam/model.sdf`)** — из сенсоров только:
- `imu_sensor` (`iris_with_standoffs::imu_link::imu_sensor`) — его `ArduPilotPlugin`
  отдаёт в SITL;
- `pilot_camera` (camera).

Никаких `<sensor type="altimeter">` / `air_pressure` / baro / GPS / magnetometer в SDF
нет.

**2. ArduPilot SITL** синтезирует барометр (а также GPS и компас) **сам, внутри
физбэкенда** — это не приходит из Gazebo. То есть на FCU барометрическая высота есть
всегда (`SIM_BARO_*`), и именно из неё сейчас берётся высота в `local_position`.

**Что это значит для плана:** пункт `EK3_SRC1_POSZ=1` (высота ← Baro) полностью валиден —
барометр доступен на уровне FCU независимо от того, что в Gazebo-модели его нет. При
переходе на ExternalNav по XY мы спокойно оставляем Z на штатном симулированном
барометре, отдельный сенсор в SDF добавлять не нужно.

Это совпадает с боевой логикой: и на Orin высоту по Z держит реальный барометр борта,
а не зрение.

---

## А компас есть, или где он должен быть?

Та же история, что с барометром: **в Gazebo-модели компаса нет, но у полётника он есть.**

**1. Gazebo-модель (`worlds/iris_cam/model.sdf`)** — сенсоры только `imu_sensor` +
`pilot_camera`. Магнитометра (`<sensor type="magnetometer">`) в SDF **нет**.

**2. ArduPilot SITL** синтезирует компас сам, внутри физбэкенда (`SIM_MAG_*`) — на
основе моделируемого положения дрона и модели магнитного поля Земли. Это не приходит
из Gazebo. То есть на FCU компас (yaw) есть всегда, и `EK3_SRC1_YAW=1` (курс ← компас)
в симе валиден.

**Где он «должен быть» — зависит от уровня:**

- *В симуляции* — добавлять магнитометр в SDF **не нужно и бесполезно**:
  `ArduPilotPlugin` отдаёт в SITL только IMU (гиро/акс), а компас/баро/GPS SITL считает
  сам. Даже если повесить `<sensor type="magnetometer">` в Gazebo, ArduPilot его не
  читает.
- *На боевом борту (Orin)* — компас это реальный аппаратный магнитометр (обычно в
  GPS-модуле или на плате FC). Ключевой нюанс проекта: **на реальном дроне без GPS
  компасу доверия мало** (помехи от рамы/моторов, металл), поэтому в боевом плане yaw
  переезжает на `EK3_SRC1_YAW=6` (курс из VINS/vision). В симе же — оставляем `=1`,
  потому что симкомпас идеальный и развязывает узел с yaw.

**Общий шаблон** (баро + компас + GPS): в Gazebo-модели только IMU+камера; баро/компас/
GPS — синтетические внутри SITL. Это и есть причина, почему `EK3_SRC1_POSZ=1` / `YAW=1`
в симе работают «бесплатно».

## Выключим GPS сейчас, а VINS ещё не поднят — GUIDED не пройдёт?

Да — и проблема глубже, чем «подождать».

### Прямой ответ

Если выключить GPS (`GPS_TYPE=0`, `POSXY=6`), а VINS ещё не выдаёт `vision_pose` — у EKF
**нет ни одного источника горизонтальной позиции**. Тогда:

- **GUIDED даже не включится.** GUIDED — позиционный режим, ArduPilot отклонит переход
  в него без оценки позиции (`arm.sh` упадёт уже на первом шаге `set_mode GUIDED`, не
  дойдя до арма).
- `ARMING_CHECK 0` тут не спасает — он убирает *проверку*, но не создаёт позицию из
  ничего.

То есть «выключить GPS, чтобы не ждать» без живого VINS = поменять «ждём GPS» на «не
взлетим вообще».

### Нюанс посерьёзнее — bootstrap монокулярного VINS (курица-яйцо)

- **VINS-Mono монокулярный → ему нужно движение для инициализации** (масштаб,
  гравитация, bias гироскопа ненаблюдаемы в покое). Стоя на земле неподвижно, VINS
  **не инициализируется** → одометрии нет → `vision_pose` не публикуется → EKF без
  позиции → не взлетим.
- Чтобы получить движение для инициализации VINS, надо взлететь. Чтобы взлететь в
  GUIDED — нужна позиция. Замкнутый круг.

**Почему сейчас это не видно:** мы летаем на GPS, и VINS инициализируется *попутно*, во
время GPS-управляемого взлёта/квадрата (движение даёт GPS). Убрав GPS, мы теряем
источник стартового движения.

### Как это реально разрывается

1. **Гибридный bootstrap (полевая процедура GPS-denied):** взлёт и раскачка в
   **ALT_HOLD** — он держит высоту по барометру и **не требует горизонтальной позиции**
   (только IMU+компас+баро, всё это есть). Поднялись, немного подвигались → VINS сошёлся
   → `vision_pose` пошёл → задаём origin → переключаемся в GUIDED/ExternalNav.
2. **Переключение источника в воздухе:** взлететь на GPS, дать VINS сойтись, затем
   сменить `EK3_SRC` на ExternalNav (ArduPilot умеет переключать наборы источников на
   лету) и только потом убирать GPS. Удобно для теста именно ветки ExternalNav в симе.
3. **Чистый «GPS off с земли сразу в GUIDED» с монокуляром — не выходит.** Это
   фундаментальное ограничение, не баг конфига.

### Вывод для плана отключения GPS

Отключение GPS — **не одномоментный flip параметра, а смена процедуры взлёта**.
Последовательность обязательна:

```
VINS поднят и СОШЁЛСЯ → vision_pose идёт → set_gp_origin → EKF healthy → GUIDED → arm → takeoff
```

а до сходимости VINS нужна фаза движения под источником, который позицию для GUIDED не
требует (ALT_HOLD) либо временно GPS.

---

## Проблема датума при ALT_HOLD-bootstrap (двигаемся до инициализации VINS)

На реальном борту GPS нет физически, поэтому bootstrap делаем только через ALT_HOLD:
взлёт и раскачка по барометру, без горизонтальной позиции. Но координаты точки взлёта
(datum, якорь №0) мы задаём руками — а пока VINS не инициализировался, дрон уже уехал
от точки старта. Возникает соблазн «двигаться чуть-чуть, чтобы уйти от датума только в
пределах погрешности». Это интуиция в правильную сторону, но проблема не совсем там.

### Развязка: датум не «уезжает», он прибит к земле

Datum — **фиксированная точка отсчёта**, а не «там, где мы окажемся, когда VINS
инициализируется». `set_gp_origin` можно (и нужно) звать **на земле, ДО взлёта** — он
не требует источника позиции, просто привязывает NED(0,0,0) к lat/lon.

```
дрон на земле в точке взлёта → set_gp_origin = datum → потом уже ALT_HOLD/раскачка
```

Раскачка двигает дрон, а не датум. Так что «двигаться чуть-чуть, чтобы не уйти от
датума» — на самом деле не про датум.

### Настоящая проблема — выравнивание кадра VINS к NED

VINS-Mono ставит свой `(0,0,0)` в точку, где он инициализировался (в воздухе, после
раскачки), и с **произвольным yaw** (относительно первого кадра, не на Север). Между
кадром VINS и датум-привязанным NED-кадром EKF есть рассогласование из двух частей:

1. **Сдвиг по позиции** = насколько дрон уехал от точки взлёта за время bootstrap
   («init drift»). В ALT_HOLD не измеряется (горизонтального источника нет). Сюда и
   попадает интуиция «двигаться чуть-чуть».
2. **Поворот по yaw** = VINS-Север ≠ настоящий Север. Это **более серьёзная** часть:
   если отдать в EKF позицию из повёрнутого кадра, а курс EKF берёт с компаса —
   измерения позиции придут «боком», и фьюжн разъедется.

Сейчас `ray_tracer` это и не делает: `offset` у него **только трансляция**, ориентация
прокинута «как есть» (в коде: `# Ориентация: пока от VINS как есть (yaw-коррекция —
отдельный шаг)`, и в `nn1_anchor_howto` это значится как «осталось»). Вопрос упирается
ровно в этот недоделанный шаг.

### Что с этим делать

**Позиционный сдвиг (часть 1) — НЕ load-bearing.** Его не нужно зануливать движением.
Архитектура держится на том, что **первая засечка NN1 обнуляет абсолютную ошибку** (весь
смысл `offset`: `vins + offset = засечка`). Поэтому:

- двигаемся **столько, сколько нужно для НАДЁЖНОЙ инициализации VINS** (монокуляру нужен
  параллакс + возбуждение, «чуть-чуть» может не хватить → плохой scale);
- остаточный сдвиг от датума принимаем как стартовую ошибку (метры);
- первая засечка NN1 её снимает.

Не жертвуй качеством init ради близости к датуму — это неправильный размен. «Чуть-чуть»
уменьшает остаток, но корректор — NN1, а не малое движение.

**Yaw (часть 2) — load-bearing, обязательно.** В момент handover (VINS сошёлся)
`ray_tracer` должен:
- снять курс с компаса в этот момент → построить yaw-поворот, выравнивающий кадр VINS к
  NED;
- применять этот поворот (а не только трансляцию) ко всем последующим `vision_pose`.

Это и есть та самая «yaw-коррекция» из TODO. По сути `offset` надо обобщить с
вектора-трансляции до полноценного SE(2)/SE(3)-преобразования VINS→NED, зафиксированного
на handover.

### Итоговая последовательность handover

```
1. на земле:    set_gp_origin = datum (якорь №0)            # датум прибит
2. ALT_HOLD:    arm → throttle вверх → раскачка             # движение для VINS
3. ждём:        /vins_estimator/odometry сошёлся, /vision_pose идёт
4. handover:    ray_tracer фиксирует T(VINS→NED):
                  yaw   = текущий компас      (часть 2, обязательно)
                  сдвиг = «дрон ≈ датум»      (часть 1, грубо; NN1 уточнит)
5. режим:       ALT_HOLD → GUIDED на ExternalNav
6. в полёте:    первая засечка NN1 снимает остаточный сдвиг
```

Датум-проблема сводится к двум вещам: **датум ставим на земле** (он не зависит от
bootstrap), а **рассогласование кадров чинит ray_tracer на handover** — критичен yaw,
позицию дочищает NN1.

> Нюанс для теста в симе: пока NN1 ещё болванка/без реф-базы, остаточный позиционный
> сдвиг никто не снимет — для отладки именно ExternalNav это терпимо (главное, чтобы
> дрон стабильно держался и летел квадрат), но «абсолютная» точность появится только с
> рабочей засечкой.
