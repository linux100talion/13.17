ИДЕЯ В ОДНУ ФРАЗУ (XXI/XXVI)
Сложность уносим в ОФЛАЙН (данные+регистрация), рантайм держим тупым: один кадр
-> топограф φ -> тонкие головы s(φ),e(φ) -> поле −∇V. БЕЗ FAISS в route-слое, без
мульти-режимного фьюза. Цена — 100 разнообразных, ХОРОШО ЗАРЕГИСТРИРОВАННЫХ
облётов и быстрый офлайн-претрейн голов под каждый маршрут.

ПОЧЕМУ ВООБЩЕ (c)-ОСНОВНОЙ (а не (a)+(b)-фьюз из XX)
100 облётов (утро/вечер/пасмурно/солнечно, ракурсы, вся карта) лечат ГЛАВНОЕ
возражение XX — хрупкость топографа = нехватка данных. На вызревшем топографе
один forward-pass защитим; рантайм-сложность (где живут отказы в полёте) уходит
в офлайн (где дёшево и безопасно). См. XXI.

================================================================================
ГЛАВНЫЙ ПРОБЕЛ — СВЯЗКА СТАДИЙ 3b/4/6 (бутылочное горло XXI = регистрация)
================================================================================
Куски стадий есть, но НЕТ оркестратора, который проводит N облётов сквозь
регистрацию в ОДИН согласованный обучающий набор. Конкретно недостаёт:

  П1. CORRESPOND-кросс (стадия 3b): поиск совпадающих МЕСТ между облётами —
      ретривал по φ (DINOv2/топограф) + геом-проверка взаимности/радиуса. Даёт
      связи local_i<->local_j там, где якорей NN1 мало. БЕЗ этого регистрация
      висит только на якорях (редких).
  П2. ОРКЕСТРАТОР (стадия 4->6): build_global_dataset — берёт N×npz (φ + лок. позы
      + якоря + кросс-связи), на каждый облёт register_to_global, сводит ВСЕ позы
      в общую рамку, выдаёт ОДИН набор (φ, ГЛОБАЛЬНАЯ xy[, якорь/облёт]) + отчёт QC
      (cross_flight_residual; битые облёты помечает). Это и есть «настоящая задача»
      из XXI.
  П3. POSE-GRAPH (стадия 5, опционально/позже): тонкая дооптимизация поверх грубой
      регистрации (GTSAM/g2o/ceres). Грубая (П2) даёт начальную точку. Можно
      отложить — на якорях+кросс-связях грубой уже хватит для первого набора.

================================================================================
ИНВАРИАНТЫ — НЕ УРОНИТЬ (XXI п.1–3, XXII)
================================================================================
  - s_filter ОСТАЁТСЯ (XXII): данные лечат помехи, НЕ фундаментальный алиасинг
    (два по-настоящему одинаковых места). И (c) memoryless -> без фильтра он
    телепортируется при привязке к VINS. Фильтр = мост (c)<->VINS, не полировка.
  - (a) сырой baseline ДЕРЖИМ тонкой НЕЗАВИСИМОЙ страховкой (OOD-детектор: топограф
    молча врёт вне 100 облётов). Гонять при низком confidence — дёшево.
  - РЕГИСТРАЦИЯ — бутылочное горло. 100 в рассогласованных рамках ХУЖЕ 10 сведённых.
    Качество меток топографа = качество регистрации. QC (стадия 8) обязателен.
  - ДВА ПРОВОДА не путать (XXIII): головы C -> (1) p̂+ковариация = ИЗМЕРЕНИЕ в
    Калман/EK3 через companion-мост (НЕ прямо в EK3); (2) поле −∇V = ЖЕЛАЕМОЕ в
    guided/offboard. FAISS/метрика — офлайн (разметка) + провод 1, НЕ в route-рантайме.

ЧЕСТНЫЕ ОГОВОРКИ (XXVI)
  - ПОКРЫТИЕ: полилиния получает метки лишь там, где облёты летали РЯДОМ (кадры с
    малым |e| по всей длине). Маршрут — в «конверте налётанного».
  - «ЗАДАТЬ МАРШРУТ» = нарисовать полилинию + переразметить позы 100 облётов + доучить
    головы (офлайн, минуты), НЕ «нарисовал-сразу-лети» как чистый геом-(b).
  - топограф нужен всё равно (вход голов = φ).
