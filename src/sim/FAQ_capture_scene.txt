================================================================================
FAQ — capture_scene.sh (атомарный прогон диагностики камеры с хоста)
================================================================================

Скрипт: src/lab/capture_scene.sh (+ src/lab/extract_frames.py).
Запускается С ХОСТА, поднимает весь стек и проходит прогон одной командой.


--------------------------------
Что делает `capture_scene.sh`?
--------------------------------

Единый АТОМАРНЫЙ прогон «от рестарта стека до результата»: сам поднимает стек,
взлетает, (облётывает или висит) с записью rosbag камеры, садится, вытаскивает
кадры по пройденному пути и заливает их на Google Drive. Это обёртка над
make-целями и скриптами src/lab — эталон «дисциплины прогона» (не ручные команды
в контейнере, а один скрипт). Назначение — диагностика того, ЧТО реально видит
камера дрона (для отладки инициализации VINS).

В отличие от `make arm`/`fly`/`land` запускается С ХОСТА из любого места:

    cd docker/sim && make capture-scene        # полный прогон с дефолтами
    # или напрямую:
    bash src/lab/capture_scene.sh


--------------------------------
Из каких фаз состоит прогон?
--------------------------------

6 фаз; каждую можно выключить флагом, дефолты = полный прогон:

    1. рестарт стека      (RESTART)   make restart-all + make wait
    2. арм + взлёт        + ОТЧЁТ      make arm ALT=… → печатает факт. z vs цель
    3. лётная фаза + bag  (RECORD)    облёт квадрата (FLY=1) ИЛИ висение (FLY=0)
    4. посадка            (LAND)      make land
    5. извлечение кадров  ПО ПУТИ     extract_frames.py: кадр каждые DIST_M метров
    6. заливка на Drive   (GDRIVE_UP) rclone

Зачем фазы: при низком RTF (≈0.05) каждая sim-секунда дорога. Чтобы дёшево
проверить «взлетает ли дрон вообще», лётную фазу режут (FLY=0 + малый
FLY_SECONDS), не трогая запись/заливку. После арма скрипт печатает фактическую z
против цели — явный ответ «взлетел/нет».


--------------------------------
Какие есть параметры (env)?
--------------------------------

| Env                      | Default                  | Что |
|--------------------------|--------------------------|-----|
| RESTART                  | 1                        | 1 = перезапуск стека (restart-all+wait); 0 = на живом стеке (⚠️ рассинхрон) |
| ALT                      | 4                        | высота взлёта, м |
| FLY                      | 1                        | 1 = облёт квадрата; 0 = висение на месте |
| SIZE / SIDE_TIME         | 5 / 8                    | геометрия квадрата (при FLY=1): сторона, м / время на сторону, с |
| FLY_SECONDS              | 55                       | длительность лётной фазы и записи bag, с (wall) |
| RECORD                   | 1                        | 1 = писать rosbag (/image_color + поза) |
| DIST_M                   | 0.5                      | шаг выборки кадров по пройденному пути, м |
| N_FRAMES                 | 30                       | макс. число кадров (0 = без лимита) |
| LAND                     | 1                        | 1 = посадка в конце |
| TOPIC                    | /image_color             | топик камеры |
| POSE_TOPIC               | /mavros/local_position/pose | поза для расчёта пути |
| CPU                      | —                        | CPU=1 → GPU-less режим (накладывает docker-compose.cpu.yml) |
| CAMERA_W / CAMERA_H      | — (1280×720 / CPU 320×180) | разрешение камеры; если задано → фаза 1 делает fresh-start |
| GDRIVE_UP                | 1                        | 1 = заливать на Drive; 0 = только снять кадры |
| GDRIVE_REMOTE / GDRIVE_DIR | gdrive / 13.17/scene_img | rclone-remote и папка на Drive |

Дефолты совпадают с прежним поведением — `make capture-scene` не изменился
(кроме выборки кадров: теперь по пути). При RECORD=1 старые rosbag'ы
(output/scene_bag*) удаляются в начале; свежий bag этого прогона (2+ ГБ)
ОСТАЁТСЯ в docker/sim/output/scene_bag для анализа.

Примеры:

    ALT=10 FLY=0 FLY_SECONDS=10 bash src/lab/capture_scene.sh   # взлёт 10м, висение, запись+заливка
    DIST_M=1.0 bash src/lab/capture_scene.sh                    # выборка реже — каждый метр пути
    GDRIVE_UP=0 bash src/lab/capture_scene.sh                   # снять кадры локально, без заливки
    RECORD=0 FLY=0 FLY_SECONDS=5 bash src/lab/capture_scene.sh  # дешёвая проверка взлёта (без bag)


--------------------------------
Как выбираются кадры — почему «по пути», а не по времени?
--------------------------------

Кадры выбираются по ПРОЙДЕННОМУ ПУТИ дрона, а не по таймеру. Это удобнее для
анализа сцены: кадры равномерны в ПРОСТРАНСТВЕ (а не во времени), число кадров
пропорционально длине маршрута.

Как работает (extract_frames.py, внутри nav-контейнера, rosbag2_py + cv_bridge):
- фаза 3 пишет в bag ДВА топика — /image_color И /mavros/local_position/pose;
- скрипт копит 3D-длину пути между позами и сохраняет: первый кадр (старт) +
  каждый раз, как с прошлого сохранения набежало >= DIST_M метров;
- имя файла несёт пройденный путь: frame_03_001.50m.jpg.

ВАЖНО: если дрон не двигался (не взлетел) или позы в bag нет → останется ТОЛЬКО
первый кадр + предупреждение. На время НЕ откатываемся.

Извлечение можно перезапустить отдельно по уже снятому bag (с другим шагом), не
делая новый прогон — внутри nav-контейнера:

    docker exec -e SCENE_DIST_M=0.5 -e SCENE_N=0 p1317_nav bash -lc \
      'source /opt/ros/humble/setup.bash; source /opt/overlay/install/setup.bash; \
       source /root/sim_ws/install/setup.bash; python3 /lab/extract_frames.py'

Env extract_frames.py: SCENE_BAG (default …/output/scene_bag), SCENE_OUT
(…/output/scene_img), SCENE_TOPIC (/image_color), SCENE_POSE
(/mavros/local_position/pose), SCENE_DIST_M (0.5), SCENE_N (30; 0 = без лимита).
Требует overlay /opt/overlay (cv_bridge против CUDA-OpenCV).


--------------------------------
Как задать разрешение камеры для прогона?
--------------------------------

Разрешение задаётся ОДНОЙ парой env — CAMERA_W / CAMERA_H, которая растекается по
всем 5 точкам автоматически (SDF-камера Gazebo, bayerizer, camera_node,
интринсики VINS, CameraInfo). docker-compose.yml интерполирует их из env хоста
(${CAMERA_W:-1280} / :-720; CPU-оверрайд → :-320 / :-180).

ПОДВОХ: env применяется при СОЗДАНИИ контейнера — restart-all (stop/start) его НЕ
перечитывает. Поэтому при заданном CAMERA_W/CAMERA_H `capture_scene.sh` в фазе 1
делает fresh-start (пересоздание контейнеров), а не restart-all. Это безопасно:
критичные SITL-параметры (SCHED_LOOP_RATE 100, ARMING_CHECK 0, DISARM_DELAY 0,
FS_GCS_ENABLE 0) лежат в host-смонтированном config/sitl-extra.parm и
применяются sim_up.sh при каждом старте.

    CAMERA_W=640 CAMERA_H=360 bash src/lab/capture_scene.sh        # 640×360 на GPU
    CPU=1 CAMERA_W=320 CAMERA_H=180 bash src/lab/capture_scene.sh  # CPU-бокс

⚠️ При RESTART=0 разрешение НЕ применится (нет пересоздания) — скрипт предупредит.


--------------------------------
Сколько идёт прогон по времени?
--------------------------------

~4–7 мин на полный прогон (дефолты). Складывается из:
  restart-all (~10–20с) + make wait до «nav: готово» (~60–120с) + прогрев EKF
  (~85с) + арм/взлёт (10–40с при успехе, до 180с при отказе) + лётная фаза
  (FLY_SECONDS=55с) + извлечение кадров (~15с) + заливка на Drive (~18с).
Разброс даёт взлёт: при зависшем takeoff ретрай до таймаута → +3 мин.
Дешёвая проверка взлёта (FLY=0, малый FLY_SECONDS) — заметно быстрее.


--------------------------------
Что нужно для заливки на Google Drive?
--------------------------------

Заливка через rclone (remote по умолчанию gdrive:). Если remote не найден —
кадры остаются локально в docker/sim/output/scene_img, скрипт выходит с ошибкой
на стадии заливки. Бокс headless → OAuth проходят в Google Cloud Shell и копируют
готовый rclone.conf на бокс. Детали — src/lab/gdrive_rclone_setup.md /
src/lab/CLAUDE.md. GDRIVE_UP=0 — пропустить заливку (снять кадры локально).
