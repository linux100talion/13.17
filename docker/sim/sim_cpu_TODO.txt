АНАЛИЗ: что реально завязано на GPU (по состоянию кода на момент ветки)
───────────────────────────────────────────────────────────────────────────
  🔴 camera_node.cpp        — весь процессинг на cv::cuda::* :
                              demosaicing (дебайер), split/merge (gain),
                              cvtColor BGR2GRAY (mono→VINS) и RGB2BGR (color).
                              ЕДИНСТВЕННАЯ жёсткая GPU-завязка в ядре.
  🔴 simulator (Gazebo)     — NVIDIA_DRIVER_CAPABILITIES=all, EGL-рендер
                              камеры дрона 1280×720. SITL сам — чистый CPU.
  🟡 nav Docker-образ       — база nvidia/cuda:12.2.2, OpenCV WITH_CUDA,
                              compose runtime: nvidia. cv:: (CPU) работают и
                              в этом образе; CUDA грузится лениво. Снять только
                              runtime: nvidia + NVIDIA_* env.
  🟢 bayerizer.py           — чистый numpy/CPU, GPU не трогает.
  🟢 VINS-Mono              — канонически CPU (LK-трекинг + Ceres).
                              TODO: подтвердить на форке linux100talion/
                              VINS-MONO-ROS2 (ветка 1317_debug) — нет cuda/GpuMat.
  ⚪ nn1/nn2 (torch/DINOv2) — GPU, но вне ядра; в этой задаче не запускаем.

ПРИНЦИПЫ
───────────────────────────────────────────────────────────────────────────
  • Все переключатели опциональны, default = текущее GPU-поведение.
  • Boot-диск/боевой Orin не трогаем — это режим только для безгпушной
    разработки в облаке/на ноуте без NVIDIA.
  • OpenCV-CUDA из образа nav НЕ выпиливаем — он просто не вызывается при
    use_cuda:=false (меньше расхождений с GPU-сборкой).
