--------------------------------------------------------------------------------
[РЕШЕНО] arm.sh: бюджет ожидания GUIDED был в wall-времени, не в sim-времени
--------------------------------------------------------------------------------
Симптом: arm выходил с `mode=STABILIZE`, `armed=False` — GUIDED не латчился.

Причина: GUIDED латчится только когда EKF дал здоровую горизонтальную позицию
(GPS-фикс + сходимость, ~15 sim-сек после старта SITL). arm.sh ждал GUIDED
циклом `for _ in seq 1 30; sleep 1` = 30 wall-сек. На GPU (RTF≈1) это ~30 sim-сек
— хватало. На CPU-боксе (RTF≈0.07) те же 30 wall-сек = ~2 sim-сек — EKF не
успевал. Поллинг «по факту» был только на стороне успеха (break по mode==GUIDED),
а give-up-бюджет остался привязан к wall-итерациям и не переносился между RTF.

Замер (с чистого старта, наблюдатель по /clock):
    wall=51s  sim=6s   gps_fix=0   mode=—
    wall=114s sim=10s  gps_fix=6   mode=—
    wall=176s sim=15s  gps_fix=6   mode=GUIDED   <- латч на sim=15s

Фикс (внесён):
  - arm.sh: бюджет ожидания GUIDED и арма считается в SIM-секундах по /clock
    (ARM_SIM_BUDGET=40, страховочный wall-потолок ARM_WALL_CAP=1200). success —
    по факту (break при mode==GUIDED / armed==true).
  - capture_scene.sh: убран фиксированный `sleep 8` прогрева EKF (команды ждут
    готовность по факту).
Проверено: на CPU-боксе FCU впервые перешёл в GUIDED (`mode=GUIDED`).

ВАЖНО — откуда взялись 40 sim-секунд (НЕ пересчитывать «обратно через RTF»):
  Бюджет ARM_SIM_BUDGET=40 я брал не из RTF, а из самого sim-времени: мы замерили,
  что EKF доходит до GPS-фикса к ~sim=10с, а GUIDED латчится к ~sim=15с. 40 = эти
  15 нужных sim-секунд + запас. RTF в этот расчёт не входит вообще.

  Следствие: 40 sim-сек — переносимая величина (свойство сходимости EKF в
  sim-времени, одинаковое на CPU и GPU). При переезде на ноут с GPU бюджет менять
  НЕ нужно — поменяется только wall-время, которое эти 15с займут (≈210 wall-сек
  на RTF 0.07 vs ≈20 на RTF 0.9). Из RTF выводился бы только страховочный
  wall-потолок ARM_WALL_CAP, но он взят с запасом (1200) под оба режима.

Примечание: takeoff.sh свой бюджет считает в 180 итерациях (~13-25 sim-сек на
RTF 0.07) — на 5-метровый набор хватает; пока не трогали. Если упрётся —
переделать так же на sim-бюджет.

[ДИАГНОСТИРОВАНО] арм режется "Arm: Gyros inconsistent" — фикс: один IMU
--------------------------------------------------------------------------------
ПРИЧИНА (поймана дословно через statustext, msgid 253 на повышенном rate, пока
arm.sh спамил арм): FCU шлёт "Arm: Gyros inconsistent" — проверка консистентности
двух симулированных IMU по гиро. Стреляет ДАЖЕ при ARMING_CHECK 0 (и сохраняется
после fresh-start, т.е. дело не в применении параметров). Force-arm (21196) не
помогает: команда ACCEPTED, но дрон не удерживается armed (на чистом стеке
armed=False стабильно) — путь A (force-arm в arm.sh) отпал.

ФИКС (внесён в config/sitl-extra.parm): оставить ОДИН IMU — сравнивать нечего:
    SIM_IMU_COUNT 1
    INS_USE2 0
    INS_USE3 0
Плюс ранее туда же FS_THR_ENABLE 0 + BRD_SAFETY_DEFLT 0 (нет RC / safety-switch).

СЛОЙ 2 — после фикса гиро всплыла следующая ОБЯЗАТЕЛЬНАЯ проверка:
"Arm: 3D Accel calibration needed" (ARMING_CHECK 0 её НЕ выключает). Параметрами
(INS_ACCOFFS/ACCSCAL) НЕ снимается: pymavlink показал, что INS_ACC_ID, scale,
offsets уже выглядят валидными, а проверка всё равно стреляет — «калибровка
выполнена» это внутреннее состояние eeprom, не параметр. Level-cal
(PREFLIGHT_CALIBRATION param5=4) его проставляет, НО: (а) принимается только когда
FCU устаканился (EKF сошёлся; иначе ACK result=1 TEMP_REJECTED); (б) вызывает
ребут FCU; (в) через MAVROS ненадёжна на низком RTF. Поэтому делаем её НАДЁЖНО и
ОДИН раз через pymavlink (прямо к SITL tcp:5762), а результат ПЕРСИСТИМ.

ФИКС (путь C — персистентный eeprom):
  - docker-compose.yml: named volume sitl_eeprom -> /root/sitl_state;
  - sim_up.sh: SITL запускается из /root/sitl_state (eeprom.bin ложится в volume,
    переживает fresh-start; --defaults применяется поверх → правки .parm работают);
  - scripts/sitl_accel_cal.py + `make sitl-cal`: ОДНОРАЗОВАЯ accel-калибровка через
    pymavlink (ждёт латч GUIDED = готовность FCU, шлёт level-cal до ACK result=0,
    проверяет что "3D Accel cal needed" ушёл). Запускать после make clean / 1-го up;
  - nav_up.sh: per-boot MAVROS-cal УБРАНА (не нужна с персистом, ребутила FCU).

СТАТУС: ✅ РЕШЕНО И ПОДТВЕРЖДЕНО. Чистый fresh-start (down/up сохраняет named
volume) → arm.sh → armed=True держится, отказов нет. Бонус: закрыт старый TODO
CLAUDE.md (SITL-параметры/калибровка теперь переживают fresh-start).

Порядок одноразовой настройки на новом боксе / после make clean:
    make CPU=1 host-setup && make CPU=1 build
    make CPU=1 fresh-start && make CPU=1 wait     # создаёт volume + свежий eeprom
    make CPU=1 sitl-cal                            # ОДИН раз: accel-cal -> eeprom
    # дальше любой fresh-start уже стартует с откалиброванным eeprom

ПОЛЁТ ✅ (arm/takeoff/hover/land на CPU РАБОТАЕТ). Два фикса по пути:
  - nav_up.sh: включён поток POSITION (set_stream_rate stream_id 6, 25 Гц) —
    без него LOCAL_POSITION_NED/GLOBAL_POSITION_INT не стримились, takeoff.sh не

--- ниже история разведки (как пришли к причине) ---
После фикса GUIDED точка отказа сместилась на сам арм.

Факты:
  - обычный /mavros/cmd/arming → success=False, result=4 (MAV_RESULT_FAILED) —
    арм ОТКЛОНЁН (не «не успел»: пробовал все 40 sim-сек, сотни попыток);
  - force-arm /mavros/cmd/command (CommandLong 400, param2=21196) →
    success=True, result=0 (MAV_RESULT_ACCEPTED);
  - armed=False прочитан после force-arm — скорее всего разрежённое чтение за
    5 wall-сек (=0.35 sim-сек), не повторный отказ. НЕ перепроверено.

Вывод: раз force-arm проходит, а обычный нет — блокирует soft-проверка ВНЕ
ARMING_CHECK. В headless-SITL без MAVProxy классические кандидаты (оба молчаливые,
ARMING_CHECK их не выключает):
  1. виртуальный safety-switch — BRD_SAFETY_DEFLT (по умолчанию «взведён»,
     обычно снимается через MAVProxy, которого у нас нет);
  2. RC/throttle failsafe — FS_THR_ENABLE (RC-вход отсутствует, нет MAVProxy/
     джойстика).
statustext с причиной поймать не удалось — на RTF 0.07 MAVROS его шлёт слишком
редко; /tmp/ArduCopter.log GCS-отказы арма не пишет.

Развилка решения (sim-only тулинг src/lab, только под SITL):
  A. Прагматично: arm.sh армит через force-arm (CommandLong 400, param2=21196).
     Гарантированно, без угадывания параметров и рестарт-циклов. В SITL force-arm
     — стандартная практика. Минус: маскирует, какая проверка блокировала.
  B. «Чисто»: найти конкретный параметр и добавить в config/sitl-extra.parm
     (host-mounted, применяется sim_up.sh при каждом старте). Кандидаты:
     BRD_SAFETY_DEFLT 0 и/или FS_THR_ENABLE 0. Минус: каждый прогон-проверка
     ~5-6 мин на CPU, возможно 2-3 итерации на угадывание.

Рекомендация: A для arm.sh (быстро разблокирует весь CPU-прогон), причину-параметр
оставить заметкой здесь. Решение за владельцем.

Следующий шаг (когда арм пройдёт): доказать takeoff 5 → hover → land на CPU,
затем полный capture_scene с записью bag и извлечением кадров.
