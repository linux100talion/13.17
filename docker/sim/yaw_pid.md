# YAW-hold (фаза 2): регулятор курса, тюнинг PID и борьба с дрейфом

Визуальный курс-холд для ALT_HOLD-бутстрапа (`src/lab/alt_hold_bootstrap.py`,
`--yaw-hold`). Держит курс дрона по камере+гироскопу, без магнитометра/GPS.
Диагностика — `liftland` (взлёт → висение → посадка) в секвенсоре
`src/lab/capture_scene.sh`; метрика — `src/lab/yaw_check.py` по ground-truth
одометрии Gazebo (`/model/iris_cam/odometry`).

## Регулятор

`alt_hold_bootstrap.py:292` (в `_on_flow_image`):

```
yf        = yaw_flow            # px/кадр ∝ визуальная yaw-СКОРОСТЬ (из FlowEstimator)
yaw_head += yf                  # ∫yf = визуальный КУРС, кламп ±imax
yu        = clip(kp*yf + ki*yaw_head, ±yaw_max)
yaw_off   = osign * blend * yu  # PWM от центра
```

Двойное прочтение (математический дуализм, `∫rate = heading`):
- **PI по yaw-СКОРОСТИ** (уставка 0, авторская рамка): `kp`=P по скорости,
  `ki`=I по скорости (∫yf=курс → «держит курс»). В этой рамке имена корректны.
- **PD по КУРСУ**: `kp`=D (демпфер yaw-rate), `ki`=P (restoring).

`yaw_flow` считается в `src/lab/flow_estimator.py:93` — медиана горизонтального
оптического потока после вычитания вращения roll+pitch (yaw гиро НЕ вычитается):
`yaw_flow = median((flow − rot_roll_pitch)[:, x])`.

## Итоги тюнинга PID (свип 2026-07-02) — ЗАКРЫТО

Полная таблица и выводы: **`src/lab/yaw_tune_results.txt`**. Свип —
`src/lab/yaw_tune_sweep.sh` (серия атомарных `liftland` + `yaw_check` + видео на Drive).

**Победитель — `kp=6, ki=0, sm=5`** (СКО 0.42° против базовых 5.19°, ×12 ровнее).
Зафиксирован в дефолтах `alt_hold_bootstrap.py` (`--yaw-kp/--yaw-ki/--yaw-smooth`).

Ключевое:
- **`ki` (restoring) ВРЕДЕН** — монотонная деградация, `ki=2` разнёс (СКО 10.65°).
  Интеграл накапливает bias `yaw_flow` → уводит истинный курс + лаг-раскачка.
  Посылка «ki держит курс» для этой петли ОПРОВЕРГНУТА. Держать `ki=0`.
- **Сглаживание нужно** (режет по-кадровый шум `yaw_flow`, который D-член усиливает):
  sm 1→3→5 = СКО 0.86→0.55→0.42. Это **временное окно N/fps**, не число кадров:
  нужность (шум) от fps не зависит, стоимость (лаг) растёт при низком fps. `sm=5`
  привязан к ~30 sim-Гц камере; на боевой (Orin, иной fps) оптимум N другой.
- **kp — оптимум 6** (U-кривая, kp=9 хуже: рост kp усиливает шум/лаг).
- **Знак `osign=+1` верный** (дрейф базы +0.04°, монотонного ухода нет).
- **Разброс метрики ~2×** между прогонами (тот же kp6/ki0/sm5: 0.42° и 0.90°) —
  мелкие различия = шум; надёжное сравнение требует усреднения ~3× на точку.
  Главный источник разброса — дрейф (см. ниже).

## Остаточная проблема: ДРЕЙФ курса от bias `yaw_flow` — В РАБОТЕ

При `ki=0` остаётся линейный **дрейф ~2–9°** (гуляет по прогонам). Это предел
точности удержания и главный источник разброса метрики. **Гейнами не лечится.**

### Диагноз (по коду)

`yaw_flow` (`flow_estimator.py:93-104`) = медиана горизонтального потока после
derotation roll+pitch. Остаток по построению = **yaw-вращение + ТРАНСЛЯЦИЯ**;
комментарий оправдывает это «в дальней сцене трансляция ≈ 0».

1. **Главный подозреваемый — утечка трансляции (монокулярная неоднозначность).**
   Средний горизонтальный поток не различает поворот по курсу и боковой снос —
   оба дают горизонтальное движение картинки. На 3 м камера смотрит вперёд-вниз
   на землю (~3 м, не «даль») → параллакс трансляции значим. Остаточная боковая
   скорость при висении (gz-hold держит позицию неидеально) → читается как yaw.
2. **Почему именно ДРЕЙФ, а не шум:** демпфер (`ki=0`) гонит `yaw_flow`→0; при
   постоянном bias `b` «ноль визуальной скорости» = ненулевая ИСТИННАЯ скорость →
   дрон крутится с постоянной истинной yaw-скоростью → линейный увод (размах≈дрейф).
3. **Roll/pitch-derotation НЕ виновата** — `R` и `rotflow_sign=+1` откалиброваны
   `flow_derotation_check` (`alt_hold_bootstrap.py:58`, остаток минимизирован).
   Смещение — от нетронутой трансляции, не от derotation.

Данные оценщику: гироскоп `/gz_imu/data_flu` (`--flow-imu-topic`), кадр
`/image_color` (`--flow-image-topic`).

### План: диагностика → фикс (не вслепую)

**Шаг 1 — измерить bias и его источник (офлайн).** Нужен bag с `/image_color` +
`/gz_imu/data_flu` + `/model/iris_cam/odometry`. Новый `src/lab/yaw_flow_check.py`:
прогоняет кадры+гиро через `FlowEstimator` → `yaw_flow(t)`, сравнивает с
- истинной yaw-скоростью (из одометрии) — масштаб/смещение;
- истинной боковой скоростью — **корреляция bias с трансляцией** (тест механизма).

Если bias коррелирует с боковой скоростью → подтверждена утечка трансляции.
(Контр-гипотезы, которые тест тоже отсеет: gyro-bias по z; ошибка масштаба fx/cx.)

**Шаг 2 — фикс, по результату.** Кандидаты:
- **(A) Washout / онлайн-оценка bias** на `yaw_flow` — убрать DC-составляющую
  rate-оценки → постоянный bias не гонит дрейф. Дёшево (мы уже `ki=0`), робастно;
  симптоматично (не убирает переменную утечку трансляции). Плата: не держит очень
  низкочастотный реальный yaw — для station-keeping приемлемо.
- **(B) Depth-aware разделение** — yaw даёт ~равномерный горизонтальный поток,
  трансляция — поле от FOE (∝1/глубина). Фитить модель вращение+трансляция, брать
  чистую вращательную часть вместо медианы. Правильно, но сложнее.
- **(C) Только «дальние»/верхние фичи** (малый параллакс) для yaw. Дёшево,
  промежуточно между A и B.

Рекомендация: начать с Шага 1 (подтвердить источник и величину bias), затем
выбрать A/B/C по данным.
