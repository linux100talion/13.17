Как работает цель `restart-all`?

Цель `restart-all` в `docker/sim/Makefile` — это **быстрый перезапуск стека без пересоздания контейнеров**.

## Что она делает

```makefile
DC := docker compose

restart-all:     ## быстрый перезапуск: stop → start (контейнеры живут, ephemeral state сохраняется)
	$(DC) stop
	$(DC) start
```

То есть две команды подряд:
1. **`docker compose stop`** — останавливает все три контейнера стека (`simulator`, `mavlink_router`, `nav`), но **не удаляет** их.
2. **`docker compose start`** — снова запускает те же самые контейнеры.

## Почему это важно (ключевое отличие от `fresh-start`)

| | `restart-all` (stop → start) | `fresh-start` (down → up) |
|---|---|---|
| Контейнеры | те же, переиспользуются | пересоздаются с нуля |
| Ephemeral state внутри контейнера | **сохраняется** | **теряется** |

Поскольку `stop/start` не уничтожает контейнеры, **выживает всё, что лежит внутри контейнера вне mounts/volumes** — в частности:
- SITL-параметры в `gazebo-iris.parm` (`SCHED_LOOP_RATE 100` и др., см. п.12 CLAUDE.md), которые иначе пришлось бы применять заново;
- установленные на лету `apt`/`pip` пакеты.

А `fresh-start` (`down → up`) контейнеры пересоздаёт — этот ephemeral state теряется (но bind mounts и named volumes `nav_colcon_build`/`nav_colcon_install` переживают и его).

## Что происходит при старте автоматически

Скрипты запуска нод (`sim_up.sh`, `nav_up.sh`) прописаны как `command:` в `docker-compose.yml`, поэтому при `start` контейнеры снова выполняют их сами — поднимается Gazebo+SITL+bridge в `simulator` и VINS+камера+MAVROS+нейросети в `nav`. Отдельно дёргать `make sim`/`make nav` не нужно.

## Как используется на практике

Это основная команда итерации разработки. Согласно «дисциплине прогона» в CLAUDE.md, стек поднимают **только целиком**:

```bash
make restart-all 2>&1 | tail -3 && make wait 2>&1
```

`make wait` после неё ждёт, пока `nav_up.sh` допишет в лог «nav: готово» (до 5 мин) — то есть пока colcon-сборка и ноды поднимутся. Ручной перезапуск отдельных процессов запрещён — состояние стека рассинхронизируется.


Что нужно сделать до запуска `restart-all`?

`restart-all` — это всего лишь `docker compose stop → start` уже **существующих** контейнеров. Поэтому она работает только если стек уже однажды поднят и есть, что перезапускать.

До первого `restart-all` нужно:

1. **`make host-setup`** — настройка хоста (нужен sudo): `xhost` для GUI Gazebo + `modprobe v4l2loopback` → `/dev/rawbayer`. Без этого камера-нода не получит устройство, а Gazebo — дисплей.

2. **`make build`** — собрать docker-образы (долго: SITL + Gazebo + OpenCV из исходников). Достаточно один раз / после изменения Dockerfile.

3. **`make up`** (или `make run`) — `docker compose up -d`, **создаёт** контейнеры. При первом подъёме `command:` сам запускает `sim_up.sh` / `nav_up.sh`.

После этого контейнеры существуют, и дальше итерация идёт через `restart-all`.

Важные нюансы:
- Если контейнеров ещё нет (после `make down`/`fresh-start`/первый раз) — `restart-all` нечего перезапускать, нужен `up`/`fresh-start`.
- По «дисциплине прогона» из CLAUDE.md стек поднимают **только целиком**, сразу с ожиданием готовности:
  ```bash
  make restart-all 2>&1 | tail -3 && make wait 2>&1
  ```
- Если переустанавливал образы или менял Dockerfile — сначала `make build`, иначе `restart-all`/`start` поднимет старый образ.

Коротко: первый раз — `host-setup → build → up`; дальше для каждого прогона — `restart-all + wait`.


Как вставлять в прогон симуляции скрипты `arm_takeoff.sh` и `fly_square.py` из `src/lab`? Как между ними переключаться?

## Их «не вставляют» в запуск стека — они уже примонтированы

`arm_takeoff.sh` и `fly_square.py` **не являются частью** `restart-all`/`sim_nav.launch.py`. Они постоянно лежат внутри `nav`-контейнера через bind mount (read-only):

```yaml
# docker-compose.yml:99
- ../../src/lab:/lab:ro     # arm/fly/land/vins_watch
```

То есть весь каталог `src/lab/` виден в контейнере как `/lab/`. Редактируешь файл на хосте → правка сразу доступна в контейнере, пересборка/перезапуск не нужны.

## Как они «вставляются в прогон» — это ручные шаги поверх поднятого стека

Стек (`restart-all`) поднимает Gazebo+SITL+VINS+MAVROS, но **дрон стоит на земле**. Дальше ты сам подаёшь команды через Makefile-цели, которые просто дёргают `docker exec` по этим скриптам:

```makefile
arm:  docker exec p1317_nav bash   /lab/arm_takeoff.sh $(ALT)
fly:  docker exec -it p1317_nav python3 /lab/fly_square.py $(FLYARGS)
land: docker exec p1317_nav bash   /lab/land.sh
```

Типичная последовательность одного прогона:

```bash
cd docker/sim
make restart-all && make wait    # поднять стек, дождаться «nav: готово»
make arm                         # arm_takeoff.sh: GUIDED → арм → взлёт 3м
make fly                         # fly_square.py: облёт квадрата (инициализация VINS)
make land                        # land.sh: посадка
```

## Это не «или-или» и не переключение — это последовательность ролей

У них **разные задачи**, они работают в связке, а не вместо друг друга:

- **`arm_takeoff.sh`** — один раз: поднять дрон в воздух (GUIDED + арм + набор высоты). Завершается сам, по факту достижения высоты.
- **`fly_square.py`** — потом: гоняет дрон по квадрату, чтобы создать параллакс и IMU excitation для инициализации VINS. Крутится, **пока не нажмёшь Ctrl+C**.

Сначала `arm`, затем `fly`. Без взлёта `fly_square.py` нечего двигать.

## «Переключение» параметрами, а не скриптами

Менять поведение — через переменные Makefile, сами скрипты не трогаешь:

```bash
make arm ALT=5                              # взлёт на 5 м
make fly FLYARGS="--size 8 --alt 4"         # квадрат 8×8, высота 4 м
make fly FLYARGS="--size 5 --side-time 5"   # быстрее, 5 с на сторону
```

Или напрямую, в обход Makefile:

```bash
docker exec p1317_nav bash /lab/arm_takeoff.sh 5
docker exec -it p1317_nav python3 /lab/fly_square.py --size 5 --alt 3 --side-time 8
```

## Если хочешь всё автоматом

`capture_scene.sh` (= `make capture-scene`) — готовая обёртка, которая сама делает `restart-all → wait → arm → fly → запись bag → land`. Это эталонный «атомарный прогон» из дисциплины CLAUDE.md; новые сценарии рекомендуется оформлять так же — обёрткой над `make`/скриптами, а не ручными командами в контейнере.

Коротко: скрипты уже «вставлены» (bind mount `/lab`), а в прогон ты их вводишь целями `make arm` → `make fly` (по порядку, не вместо друг друга); поведение настраивается `ALT=`/`FLYARGS=`.


Что делает `vins_watch.sh` и как им пользоваться?

`vins_watch.sh` — это монитор состояния VINS в реальном времени. Сам он ничем не управляет (не армит, не летит), только **наблюдает**: помогает понять, инициализировался ли VINS и нет ли характерных ошибок. Лежит в `src/lab/`, примонтирован в `nav`-контейнер как `/lab/vins_watch.sh` (bind mount `/lab:ro`).

## Что он показывает

Делает две вещи параллельно:

1. **Фильтрует лог** `/root/sim_ws/output/sim_nav.log` (`tail -f` + `grep`) по ключевым событиям VINS:
   `Initialization`, `NON_LINEAR`, `disorder`, `unstable`, `reboot`, `failure`, `IMU excitation`, `features`, `WARN`, `ERROR`.
2. **Меряет частоту** топика `/vins_estimator/odometry` (`ros2 topic hz --window 10`, строки с префиксом `[odom]`).

То есть в одном окне видно и текстовые события инициализации/ошибок, и идёт ли одометрия (и с какой частотой).

## Как запускать

```bash
make vins-watch
# или напрямую:
docker exec -it p1317_nav bash /lab/vins_watch.sh
```

Выход — `Ctrl+C` (фоновый замер `ros2 topic hz` при выходе убивается).

## Как его место в прогоне

Это диагностический наблюдатель — его держат в отдельном окне, пока в другом идёт полёт. Типовой сценарий инициализации VINS:

```
[окно 1]  make vins-watch        # смотреть лог + частоту одометрии
[окно 2]  make arm && make fly    # взлететь и пойти по квадрату
```

На что смотреть:
- ждём `Initialization finish!` → затем `solver_flag: NON_LINEAR` — VINS инициализировался;
- после инициализации `[odom]` начинает показывать ненулевую частоту `/vins_estimator/odometry`;
- НЕ должно быть `numerical unstable`, `imu message in disorder`, `system reboot!` — это признаки проблем (см. п.11 CLAUDE.md про IMU timesync; при `reboot` — `make land`, затем `make arm && make fly` заново).

Важно: `vins_watch.sh` только читает лог и топик — запускать/перезапускать его можно сколько угодно, на состояние стека он не влияет (под «дисциплину прогона» это разрешённое «только смотреть»).


Что делает `land.sh` и как им пользоваться?

`land.sh` — это посадка дрона: переводит FCU в режим **LAND**, после чего ArduPilot сам снижается и садится на месте (по вертикали, без горизонтального перемещения). Лежит в `src/lab/`, примонтирован в `nav`-контейнер как `/lab/land.sh`.

## Что он делает под капотом

Один вызов MAVROS-сервиса смены режима:

```bash
ros2 service call /mavros/set_mode mavros_msgs/srv/SetMode '{custom_mode: "LAND"}'
```

Скрипт печатает `mode_sent: true`, если команда принята. Дальше снижением управляет сам автопилот — скрипт ничего не ждёт и не поллит высоту, он просто отдаёт команду и завершается.

## Как запускать

```bash
make land
# или напрямую:
docker exec p1317_nav bash /lab/land.sh
```

## Место в прогоне

`land.sh` — это завершение лётной части прогона, противоположность `arm_takeoff.sh`. Типовая последовательность:

```bash
make arm     # взлёт
make fly     # облёт квадрата (Ctrl+C, чтобы остановить)
make land    # посадка
```

Когда применять:
- штатно — после `make fly` в конце прогона;
- при потере VINS трекинга (`system reboot!` в `vins_watch.sh`) — остановить `fly`, сделать `make land`, затем `make arm && make fly` заново (см. «Известные ограничения» в `src/lab/CLAUDE.md`).

Замечания:
- `land.sh` шлёт только команду режима — он НЕ дизармит явно. После касания земли ArduPilot дизармит сам (в SITL с `DISARM_DELAY 0` — сразу).
- `fly_square.py` к этому моменту должен быть остановлен (Ctrl+C): иначе он продолжает слать setpoint'ы и может перебивать снижение.
