--------------------------------
Как работает цель `restart-all`?
--------------------------------

Цель `restart-all` в `docker/sim/Makefile` — это **быстрый перезапуск стека без пересоздания контейнеров**.

## Что она делает

```makefile
DC := docker compose

restart-all:     ## быстрый перезапуск: stop → start (контейнеры живут, ephemeral state сохраняется)
	$(DC) stop
	$(DC) start
```

То есть две команды подряд:
1. **`docker compose stop`** — останавливает все три контейнера стека (`simulator`, `mavlink_router`, `nav`), но **не удаляет** их.
2. **`docker compose start`** — снова запускает те же самые контейнеры.

## Почему это важно (ключевое отличие от `fresh-start`)

| | `restart-all` (stop → start) | `fresh-start` (down → up) |
|---|---|---|
| Контейнеры | те же, переиспользуются | пересоздаются с нуля |
| Ephemeral state внутри контейнера | **сохраняется** | **теряется** |

Поскольку `stop/start` не уничтожает контейнеры, **выживает всё, что лежит внутри контейнера вне mounts/volumes** — в частности:
- SITL-параметры в `gazebo-iris.parm` (`SCHED_LOOP_RATE 100` и др., см. п.12 CLAUDE.md), которые иначе пришлось бы применять заново;
- установленные на лету `apt`/`pip` пакеты.

А `fresh-start` (`down → up`) контейнеры пересоздаёт — этот ephemeral state теряется (но bind mounts и named volumes `nav_colcon_build`/`nav_colcon_install` переживают и его).

## Что происходит при старте автоматически

Скрипты запуска нод (`sim_up.sh`, `nav_up.sh`) прописаны как `command:` в `docker-compose.yml`, поэтому при `start` контейнеры снова выполняют их сами — поднимается Gazebo+SITL+bridge в `simulator` и VINS+камера+MAVROS+нейросети в `nav`. Отдельно дёргать `make sim`/`make nav` не нужно.

## Как используется на практике

Это основная команда итерации разработки. Согласно «дисциплине прогона» в CLAUDE.md, стек поднимают **только целиком**:

```bash
make restart-all 2>&1 | tail -3 && make wait 2>&1
```

`make wait` после неё ждёт, пока `nav_up.sh` допишет в лог «nav: готово» (до 5 мин) — то есть пока colcon-сборка и ноды поднимутся. Ручной перезапуск отдельных процессов запрещён — состояние стека рассинхронизируется.


--------------------------------
Что нужно сделать до запуска `restart-all`?
--------------------------------

`restart-all` — это всего лишь `docker compose stop → start` уже **существующих** контейнеров. Поэтому она работает только если стек уже однажды поднят и есть, что перезапускать.

До первого `restart-all` нужно:

1. **`make host-setup`** — настройка хоста (нужен sudo): `xhost` для GUI Gazebo + `modprobe v4l2loopback` → `/dev/rawbayer`. Без этого камера-нода не получит устройство, а Gazebo — дисплей.

2. **`make build`** — собрать docker-образы (долго: SITL + Gazebo + OpenCV из исходников). Достаточно один раз / после изменения Dockerfile.

3. **`make up`** (или `make run`) — `docker compose up -d`, **создаёт** контейнеры. При первом подъёме `command:` сам запускает `sim_up.sh` / `nav_up.sh`.

После этого контейнеры существуют, и дальше итерация идёт через `restart-all`.

Важные нюансы:
- Если контейнеров ещё нет (после `make down`/`fresh-start`/первый раз) — `restart-all` нечего перезапускать, нужен `up`/`fresh-start`.
- По «дисциплине прогона» из CLAUDE.md стек поднимают **только целиком**, сразу с ожиданием готовности:
  ```bash
  make restart-all 2>&1 | tail -3 && make wait 2>&1
  ```
- Если переустанавливал образы или менял Dockerfile — сначала `make build`, иначе `restart-all`/`start` поднимет старый образ.

Коротко: первый раз — `host-setup → build → up`; дальше для каждого прогона — `restart-all + wait`.


--------------------------------
Как вставлять в прогон симуляции скрипты `arm_takeoff.sh` и `fly_square.py` из `src/lab`? Как между ними переключаться?
--------------------------------

## Их «не вставляют» в запуск стека — они уже примонтированы

`arm_takeoff.sh` и `fly_square.py` **не являются частью** `restart-all`/`sim_nav.launch.py`. Они постоянно лежат внутри `nav`-контейнера через bind mount (read-only):

```yaml
# docker-compose.yml:99
- ../../src/lab:/lab:ro     # arm/fly/land/vins_watch
```

То есть весь каталог `src/lab/` виден в контейнере как `/lab/`. Редактируешь файл на хосте → правка сразу доступна в контейнере, пересборка/перезапуск не нужны.

## Как они «вставляются в прогон» — это ручные шаги поверх поднятого стека

Стек (`restart-all`) поднимает Gazebo+SITL+VINS+MAVROS, но **дрон стоит на земле**. Дальше ты сам подаёшь команды через Makefile-цели, которые просто дёргают `docker exec` по этим скриптам:

```makefile
arm:  docker exec p1317_nav bash   /lab/arm_takeoff.sh $(ALT)
fly:  docker exec -it p1317_nav python3 /lab/fly_square.py $(FLYARGS)
land: docker exec p1317_nav bash   /lab/land.sh
```

Типичная последовательность одного прогона:

```bash
cd docker/sim
make restart-all && make wait    # поднять стек, дождаться «nav: готово»
make arm                         # arm_takeoff.sh: GUIDED → арм → взлёт 3м
make fly                         # fly_square.py: облёт квадрата (инициализация VINS)
make land                        # land.sh: посадка
```

## Это не «или-или» и не переключение — это последовательность ролей

У них **разные задачи**, они работают в связке, а не вместо друг друга:

- **`arm_takeoff.sh`** — один раз: поднять дрон в воздух (GUIDED + арм + набор высоты). Завершается сам, по факту достижения высоты.
- **`fly_square.py`** — потом: гоняет дрон по квадрату, чтобы создать параллакс и IMU excitation для инициализации VINS. Крутится, **пока не нажмёшь Ctrl+C**.

Сначала `arm`, затем `fly`. Без взлёта `fly_square.py` нечего двигать.

## «Переключение» параметрами, а не скриптами

Менять поведение — через переменные Makefile, сами скрипты не трогаешь:

```bash
make arm ALT=5                              # взлёт на 5 м
make fly FLYARGS="--size 8 --alt 4"         # квадрат 8×8, высота 4 м
make fly FLYARGS="--size 5 --side-time 5"   # быстрее, 5 с на сторону
```

Или напрямую, в обход Makefile:

```bash
docker exec p1317_nav bash /lab/arm_takeoff.sh 5
docker exec -it p1317_nav python3 /lab/fly_square.py --size 5 --alt 3 --side-time 8
```

## Если хочешь всё автоматом

`capture_scene.sh` (= `make capture-scene`) — готовая обёртка, которая сама делает `restart-all → wait → arm → fly → запись bag → land`. Это эталонный «атомарный прогон» из дисциплины CLAUDE.md; новые сценарии рекомендуется оформлять так же — обёрткой над `make`/скриптами, а не ручными командами в контейнере.

Коротко: скрипты уже «вставлены» (bind mount `/lab`), а в прогон ты их вводишь целями `make arm` → `make fly` (по порядку, не вместо друг друга); поведение настраивается `ALT=`/`FLYARGS=`.


--------------------------------
Что делает `vins_watch.sh` и как им пользоваться?
--------------------------------

`vins_watch.sh` — это монитор состояния VINS в реальном времени. Сам он ничем не управляет (не армит, не летит), только **наблюдает**: помогает понять, инициализировался ли VINS и нет ли характерных ошибок. Лежит в `src/lab/`, примонтирован в `nav`-контейнер как `/lab/vins_watch.sh` (bind mount `/lab:ro`).

## Что он показывает

Делает две вещи параллельно:

1. **Фильтрует лог** `/root/sim_ws/output/sim_nav.log` (`tail -f` + `grep`) по ключевым событиям VINS:
   `Initialization`, `NON_LINEAR`, `disorder`, `unstable`, `reboot`, `failure`, `IMU excitation`, `features`, `WARN`, `ERROR`.
2. **Меряет частоту** топика `/vins_estimator/odometry` (`ros2 topic hz --window 10`, строки с префиксом `[odom]`).

То есть в одном окне видно и текстовые события инициализации/ошибок, и идёт ли одометрия (и с какой частотой).

## Как запускать

```bash
make vins-watch
# или напрямую:
docker exec -it p1317_nav bash /lab/vins_watch.sh
```

Выход — `Ctrl+C` (фоновый замер `ros2 topic hz` при выходе убивается).

## Как его место в прогоне

Это диагностический наблюдатель — его держат в отдельном окне, пока в другом идёт полёт. Типовой сценарий инициализации VINS:

```
[окно 1]  make vins-watch        # смотреть лог + частоту одометрии
[окно 2]  make arm && make fly    # взлететь и пойти по квадрату
```

На что смотреть:
- ждём `Initialization finish!` → затем `solver_flag: NON_LINEAR` — VINS инициализировался;
- после инициализации `[odom]` начинает показывать ненулевую частоту `/vins_estimator/odometry`;
- НЕ должно быть `numerical unstable`, `imu message in disorder`, `system reboot!` — это признаки проблем (см. п.11 CLAUDE.md про IMU timesync; при `reboot` — `make land`, затем `make arm && make fly` заново).

Важно: `vins_watch.sh` только читает лог и топик — запускать/перезапускать его можно сколько угодно, на состояние стека он не влияет (под «дисциплину прогона» это разрешённое «только смотреть»).


--------------------------------
Что делает `land.sh` и как им пользоваться?
--------------------------------

`land.sh` — это посадка дрона: переводит FCU в режим **LAND**, после чего ArduPilot сам снижается и садится на месте (по вертикали, без горизонтального перемещения). Лежит в `src/lab/`, примонтирован в `nav`-контейнер как `/lab/land.sh`.

## Что он делает под капотом

Один вызов MAVROS-сервиса смены режима:

```bash
ros2 service call /mavros/set_mode mavros_msgs/srv/SetMode '{custom_mode: "LAND"}'
```

Скрипт печатает `mode_sent: true`, если команда принята. Дальше снижением управляет сам автопилот — скрипт ничего не ждёт и не поллит высоту, он просто отдаёт команду и завершается.

## Как запускать

```bash
make land
# или напрямую:
docker exec p1317_nav bash /lab/land.sh
```

## Место в прогоне

`land.sh` — это завершение лётной части прогона, противоположность `arm_takeoff.sh`. Типовая последовательность:

```bash
make arm     # взлёт
make fly     # облёт квадрата (Ctrl+C, чтобы остановить)
make land    # посадка
```

Когда применять:
- штатно — после `make fly` в конце прогона;
- при потере VINS трекинга (`system reboot!` в `vins_watch.sh`) — остановить `fly`, сделать `make land`, затем `make arm && make fly` заново (см. «Известные ограничения» в `src/lab/CLAUDE.md`).

Замечания:
- `land.sh` шлёт только команду режима — он НЕ дизармит явно. После касания земли ArduPilot дизармит сам (в SITL с `DISARM_DELAY 0` — сразу).
- `fly_square.py` к этому моменту должен быть остановлен (Ctrl+C): иначе он продолжает слать setpoint'ы и может перебивать снижение.


--------------------------------
Что делает `extract_frames.py` и как им пользоваться?
--------------------------------

`extract_frames.py` — это пост-обработка записанного rosbag: вытаскивает из него N кадров камеры с равным шагом по времени и сохраняет их в JPEG. Это шаг 5 в `capture_scene.sh` — обычно отдельно его дёргать не нужно, но можно перегнать уже снятый bag в картинки повторно (с другим N/шагом), не делая новый прогон.

## Что он делает

Читает rosbag (`rosbag2_py.SequentialReader`, storage `sqlite3`), идёт по сообщениям топика `/image_color`, и **с шагом `STEP_NS` по времени бага** декодит кадр через `cv_bridge` (`bgr8`) и пишет `frame_NN.jpg` (JPEG quality 90), пока не наберёт `N` кадров. Перед записью чистит старые `*.jpg` в выходной папке.

## Где запускается

**Только ВНУТРИ `nav`-контейнера** — ему нужен `cv_bridge` из overlay `/opt/overlay` (собранный против нашего OpenCV; см. CLAUDE.md про ABI). С хоста напрямую не пойдёт.

```bash
# обычно — автоматически внутри capture_scene.sh (шаг 5/6).
# вручную по уже снятому output/scene_bag:
docker exec \
  -e SCENE_N=30 -e SCENE_STEP_NS=1000000000 -e SCENE_TOPIC=/image_color \
  p1317_nav bash -lc \
  'source /opt/ros/humble/setup.bash; source /opt/overlay/install/setup.bash; source /root/sim_ws/install/setup.bash; python3 /lab/extract_frames.py'
```

## Параметры (через env)

| Env | Default | Что |
|---|---|---|
| `SCENE_BAG` | `/root/sim_ws/output/scene_bag` | путь к rosbag (внутри контейнера) |
| `SCENE_OUT` | `/root/sim_ws/output/scene_img` | куда писать JPEG |
| `SCENE_TOPIC` | `/image_color` | топик изображения |
| `SCENE_N` | 30 | сколько кадров |
| `SCENE_STEP_NS` | 1000000000 | шаг между кадрами, нс (1с) |

Пути внутри контейнера, но `output/` смонтирован на хост (`docker/sim/output/`), так что bag читается и кадры появляются прямо в `output/scene_img` на хосте.

## Замечания

- Шаг считается **по timestamp'ам бага** (`t`), а не по wall-time, и от первого встреченного кадра топика. При низком RTF (CPU-бокс) 1 с шага = 1 с sim-времени — кадры всё равно идут «равномерно по сцене».
- Если кадров в баге меньше, чем `N` (короткий `FLY_SECONDS`), сохранится сколько есть — в конце печатает фактическое число.
- Декод именно в `bgr8` — это полноцветный выход `camera_node` (`/image_color`), тот же, что идёт в nav-сторону (nn1/nn2).


--------------------------------
Что делает `arm_takeoff.sh` и как им пользоваться?
--------------------------------

`arm_takeoff.sh` — это взлёт дрона: переводит FCU в режим **GUIDED**, армирует моторы и поднимает на заданную высоту. Это первый лётный шаг прогона (противоположность `land.sh`). Лежит в `src/lab/`, примонтирован в `nav`-контейнер как `/lab/arm_takeoff.sh`.

## Что он делает (3 фазы, каждая — по факту, не по `sleep`)

1. **GUIDED** — вызывает `/mavros/set_mode`, затем поллит `/mavros/state` поле `mode`, пока не станет `GUIDED` (до 30 попыток).
2. **Арм** — в цикле дёргает `/mavros/cmd/arming {value: true}` и проверяет `/mavros/state` поле `armed`, пока не станет `true` (до 30 попыток с ретраями).
3. **Взлёт** — вызывает `/mavros/cmd/takeoff` на `ALT`, потом поллит фактическую высоту `/mavros/local_position/pose` (`pose.position.z`), пока не достигнет **95 % от `ALT`** (до 180 попыток).

## ⏱ Почему поллинг, а не фиксированные `sleep`

Ключевой момент: скрипт ждёт **по факту** (опрос `mode`/`armed`/`z`), а не фиксированными паузами. Симуляция идёт с низким RTF (на CPU-боксе ~13× медленнее реального), поэтому `sleep 5` соответствовал бы долям sim-секунды — дрон не успел бы ни сменить режим, ни набрать высоту, и «взлёт» завершался бы у земли. Здесь `sleep` — это лишь **интервал опроса** (wall), а не предположение о длительности. Поэтому скрипт RTF-независим. Поллинг высоты заодно гарантирует, что EKF получил origin к моменту запуска `fly_square.py`.

## Как запускать

```bash
make arm            # взлёт на 3 м (ALT по умолчанию)
make arm ALT=5      # взлёт на 5 м
# или напрямую:
docker exec p1317_nav bash /lab/arm_takeoff.sh 5
```

Высота передаётся первым позиционным аргументом (через `make` — переменной `ALT`).

## Место в прогоне

```bash
make restart-all && make wait    # стек поднят, дрон на земле
make arm                         # ← GUIDED + арм + взлёт
make fly                         # дальше — облёт квадрата
```

## Замечания

- Требует **`ARMING_CHECK 0`** в SITL (задано в `config/sitl-extra.parm`) — иначе арм отклонит проверками.
- Завершается сам по достижении высоты; если взлёт «завис», ретраит до таймаута (фаза высоты — до 180 циклов опроса), из-за чего весь прогон может растянуться (см. бюджет времени в `src/lab/CLAUDE.md`).
- Раньше требовалась ручная пауза 3–5 с после арма (ждать EKF origin) — теперь не нужна: поллинг высоты доходит до набора уже после установки origin.


--------------------------------
Что делает `fly_square.py` и как им пользоваться?
--------------------------------

`fly_square.py` — это непрерывный облёт квадрата для **прогрева/инициализации VINS**. VINS не инициализируется, пока дрон стоит: ему нужны параллакс (смена ракурса) и IMU excitation (ускорения). Этот скрипт гоняет дрон по квадрату, создавая и то, и другое. Это второй лётный шаг (после `arm_takeoff.sh`). Лежит в `src/lab/`, примонтирован как `/lab/fly_square.py`.

## Что он делает

Это ROS2-нода (`fly_square`), которая публикует целевую позу в `/mavros/setpoint_position/local` на **10 Гц** (frame `map`). Квадрат — 4 угла относительно старта (`(0,0)→(1,0)→(1,1)→(0,1)`, умноженные на `--size`), высота держится `--alt`. Каждую сторону проходит за `--side-time` секунд, потом переходит к следующему углу — и так по кругу, **пока не прервёшь Ctrl+C**.

## ⏱ Работает на sim-времени

Нода сама ставит `use_sim_time=True` и сажается на `/clock`: таймер, отсчёт сторон и штамп setpoint — всё в sim-часах. Без этого при низком RTF (CPU-бокс, ~13× медленнее) сторона квадрата «прошла» бы за доли sim-секунды, дрон не успевал бы лететь, а штамп setpoint разъезжался бы с sim-временем FCU. Поэтому `--side-time` — это секунды **SIM**-времени: на низком RTF квадрат проходится корректно в sim-пространстве, просто дольше по реальным часам. Паблиш 10 sim-Гц — для GUIDED-таргета достаточно.

## Как запускать

```bash
make fly                                     # квадрат 5×5 м, высота 3 м, 8 с/сторона
make fly FLYARGS="--size 8 --alt 4"          # квадрат 8×8 м, высота 4 м
make fly FLYARGS="--size 5 --side-time 5"    # быстрее — 5 с на сторону
# или напрямую:
docker exec -it p1317_nav python3 /lab/fly_square.py --size 5 --alt 3 --side-time 8
```

Выход — `Ctrl+C` (нода печатает «Остановлено»).

| Параметр | Default | Что |
|---|---|---|
| `--size` | 5.0 | сторона квадрата, м |
| `--alt` | 3.0 | высота полёта, м |
| `--side-time` | 8.0 | время на сторону, с (SIM-время) |

## Место в прогоне

```bash
make arm                  # сначала взлёт (нужен EKF origin для local-координат)
make fly                  # ← облёт квадрата, держим до инициализации VINS
# в соседнем окне: make vins-watch — ждём "Initialization finish!" → "NON_LINEAR"
make land                 # Ctrl+C по fly, затем посадка
```

## Замечания

- Использует **локальные координаты** (`map` frame) → требует EKF origin. Поэтому строго **после** `arm_takeoff.sh` (тот доводит до набора высоты, к этому моменту origin есть).
- При потере трекинга (`system reboot!` в `vins_watch.sh`) — остановить `fly`, `make land`, затем `make arm && make fly` заново.
- Перед `make land` облёт надо остановить (Ctrl+C): иначе нода продолжает слать setpoint'ы и перебивает снижение. В `capture_scene.sh` это делается автоматически (`pkill -f fly_square.py`).


--------------------------------
Что делает байеризатор (`bayerizer.py`) и зачем он нужен?
--------------------------------

`bayerizer.py` (`src/sim/bayerizer.py`) — это **мост из Gazebo в виртуальную Bayer-камеру**. Его задача — позволить боевой `camera_node` работать в симуляции **БЕЗ переписывания**: камера-нода рассчитана на реальный сенсор, который отдаёт сырой Bayer-мозаик, а Gazebo отдаёт готовый RGB. Байеризатор «портит» RGB обратно до Bayer — как сделал бы настоящий сенсор — чтобы дальше всё шло штатным путём.

## Зачем он вообще нужен

Боевая `camera_node` делает: V4L2-захват сырого Bayer → CUDA-дебайер → `/image_mono` (+ `/image_color`). В симуляции реального сенсора нет, а есть RGB из Gazebo. Чтобы не плодить отдельную «симуляционную» камеру-ноду, мы разворачиваем картинку:

```
Gazebo /camera/image_raw (RGB)
  → bayerizer.py  (RGB → Bayer16)
  → /dev/rawbayer (v4l2loopback, модуль ядра ХОСТА)
  → camera_node (device:=/dev/rawbayer) → CUDA-дебайер обратно → /image_mono + /image_color
```

В камера-ноде меняется **только** параметр `device:=/dev/rawbayer` — весь остальной боевой код (дебайер, штампы, топики) тот же.

## Что он делает по шагам

1. Подписан на `/camera/image_raw` (ROS, событийно — пишет ровно по приходу кадра, своего fps нет).
2. Раскладывает RGB-кадр в однослойный Bayer-мозаик по паттерну (`pattern:=GRBG` по умолчанию; поддержаны GRBG/RGGB/BGGR/GBRG) — `_mosaic()` берёт нужный цвет в каждой клетке сетки 2×2.
3. Пакует 8-битный Bayer в младший байт `uint16` (camera_node читает кадр как `CV_16UC1` и делает NORM_MINMAX 16→8).
4. Пишет в `/dev/rawbayer`.

## Почему такой странный транспорт (BGR4, ctypes, нулевой кадр)

v4l2loopback капризен — все эти детали выстраданы при отладке на GCE (см. п.4 раздела симуляции в корневом CLAUDE.md):
- **Только BGR4** (32-бит BGRA): BA10/Y16 → EINVAL. Поэтому 16-битные Bayer-данные едут «контрабандой» внутри BGR4-буфера: кадр = `width*height*2` байт bayer16 + нули-паддинг до `sizeimage`. camera_node берёт первые `w*h*2` байт как `CV_16UC1`.
- **`v4l2_format` = 208 байт** (не 204) — на x86_64 есть 4 байта padding из-за выравнивания указателя в union. Поэтому формат собирается через `ctypes` с `_align = c_uint64`.
- **`ready_for_capture` ставится только после первого `write()`** — поэтому байеризатор открывает устройство и пишет один нулевой кадр прямо в `__init__()` (eager init). До этого camera_node получал бы EINVAL на G_FMT(CAPTURE).

## Как он запускается

**Не из `sim_nav.launch.py`**, а отдельным `nohup`-процессом в `scripts/nav_up.sh`. Причина: если байеризатор падает внутри launch, launch-менеджер убил бы всё дерево (camera_node + VINS). Вне launch он изолирован. camera_node при этом стартует с задержкой (`TimerAction` ~3–4 с), чтобы байеризатор успел активировать capture-сторону loopback.

Перед запуском на **хосте** должен быть поднят v4l2loopback (`make host-setup`: `modprobe v4l2loopback` + симлинк `/dev/rawbayer`), устройство пробрасывается в контейнер `nav` через `devices:`.

## Разрешение

Размеры задаются параметрами `width`/`height` (из env `CAMERA_W`/`CAMERA_H` в `nav_up.sh`) и должны совпадать с камерой Gazebo и camera_node. Default 1280×720; CPU-оверрайд (`docker-compose.cpu.yml`) — 320×180. Разрешение завязано в 5 точках одним env (см. `src/sim/CLAUDE.md`).

## fps и время

Байеризатор fps не навязывает — он событийный, а camera_node читает блокирующим `fread`, так что кадры идут 1:1 с Gazebo (без дублей/дропов). При этом `/dev/rawbayer` — это кусок на **wall-clock** (у ядра нет sim-времени); согласованность по времени восстанавливает camera_node, перештамповывая каждый кадр `get_clock()->now()` (sim-время). Подробно про два времени и этот стык — `src/sim/CLAUDE.md`.


--------------------------------
Что такое `sim_nav.launch.py` и что он запускает?
--------------------------------

`sim_nav.launch.py` (`src/sim/sim_nav.launch.py`) — это launch-файл **nav-стороны в симуляции**. Он поднимает камеру-ноду, VINS и нейросети, причём всем ставит `use_sim_time:=true` — чтобы они брали время из `/clock` (его публикует `ros_gz_bridge` в контейнере `simulator`). Без этого штампы кадров/IMU разъедутся с симуляцией и VINS будет молча расходиться. Запускается внутри контейнера `nav` (из `nav_up.sh` после colcon build).

## Что он запускает

`TimerAction(period=4с)` — даёт байеризатору активировать `/dev/rawbayer`, затем поднимает с задержкой:
1. **camera_node** — `/dev/rawbayer` → `/image_mono` (VINS) + `/image_color` (nav). Параметры `device`, `width`, `height`.
2. **feature_tracker** (VINS) — с конфигом `config_file`.
3. **vins_estimator** (VINS) — с тем же конфигом + remap `/feature_tracker/feature → /feature`, `/restart`.

Плюс через `IncludeLaunchDescription` подтягивает **`nav.launch.py`** из `nav_pkg` (nn1_anchor + nn2_scene + openhd_streamer), тоже с `use_sim_time:=true`.

## Чего он НЕ запускает (важно)

- **`ros_gz_bridge`** — он ИСТОЧНИК `/clock`, ему `use_sim_time` ставить нельзя; живёт в контейнере `simulator`.
- **MAVROS** — свой launch, `use_sim_time` для него настраивается отдельно (тонкий момент с IMU-штампами от FCU, см. CLAUDE.md п.11).
- **Байеризатор** — намеренно ВНЕ этого launch, отдельным `nohup` в `nav_up.sh`. Причина: если бы байеризатор падал внутри launch, launch-менеджер убил бы всё дерево (camera_node + VINS). Отсюда же `TimerAction(4с)`: `nav_up.sh` сначала поднимает байеризатор и ждёт `/dev/rawbayer`, потом зовёт этот launch.

## Два переключателя через env (не правя launch)

- **`CAMERA_NODE`** — какой executable камеры: `camera_node` (боевой CUDA-дебайер, default) или `camera_node_cpu` (drop-in CPU-дебайер для машин без GPU). CPU-оверрайд compose выставляет `camera_node_cpu`.
- **`CAMERA_W`/`CAMERA_H`** — разрешение камеры (default 1280×720; CPU-режим 320×180). Функция `_vins_config()` при не-дефолтном разрешении масштабирует `image_width/height` и интринсики `fx/fy/cx/cy` из ОДНОГО источника (`sim.yaml`) в `/tmp/sim_WxH.yaml` — чтобы не держать второй .yaml, который бы дрейфовал. На 1280×720 возвращает `sim.yaml` как есть (боевой/GPU путь не трогаем).

## Как запускается

Обычно — автоматически из `scripts/nav_up.sh`. Вручную (в контейнере `nav`, после colcon build):

```bash
ros2 launch /root/sim_ws/src/sim/sim_nav.launch.py
```

Под «дисциплину прогона» отдельно его дёргать не стоит — стек поднимают целиком через `make restart-all` (см. выше).


--------------------------------
Что делает `nav_up.sh` и как он запускается?
--------------------------------

`nav_up.sh` (`docker/sim/scripts/nav_up.sh`) — это **entrypoint nav-стороны**: один скрипт, который внутри контейнера `nav` приводит окружение в порядок, собирает workspace и поднимает все ноды (байеризатор → camera_node + VINS → MAVROS). Это он пишет в лог финальное «nav: готово», которого ждёт `make wait`.

## Как запускается

- **Автоматически** — прописан как `command:` контейнера `nav` в `docker-compose.yml`, поэтому стартует сам при `up`/`start`/`restart-all`.
- **Вручную** — `make nav` (= `docker exec -i p1317_nav bash -s < scripts/nav_up.sh`). Он смонтирован как `/scripts/nav_up.sh:ro`, так что правки на хосте применяются без пересборки образа.

## Что он делает по шагам

1. **Source окружения по порядку** (важно!): ROS Humble → overlay `/opt/overlay` (там `cv_bridge`, собранный против CUDA-OpenCV) → workspace. Неверный порядок = ABI-краш cv_bridge.
2. **Проверки/починка зависимостей** (идемпотентно, переживает `fresh-start`, где apt/pip-пакеты теряются):
   - `vins_oss` смонтирован (bind mount из `/root/VINS-MONO-ROS2`, ветка `1317_debug`) — иначе ошибка;
   - `image_transport` есть (нужен feature_tracker/pose_graph) — fallback на apt, если образ старый;
   - `numpy<2` — cv_bridge собран против NumPy 1.x, NumPy 2.x ломает ABI; даунгрейд при необходимости.
3. **colcon build** — только если workspace ещё не собран (`install/setup.bash` нет). Для принудительной пересборки — `make nav-rebuild`.
4. **Байеризатор** (`nohup`, ВНЕ launch) — Gazebo RGB → `/dev/rawbayer`; затем **ждёт до 60 с**, пока `v4l2-ctl` увидит формат (eager-init активировал capture-сторону loopback). Разрешение из `CAMERA_W`/`CAMERA_H`.
5. **`sim_nav.launch.py`** (`nohup`) — camera_node + feature_tracker + vins_estimator + nav_pkg, все с `use_sim_time`.
6. **MAVROS** (`nohup`, отдельно) — вход от mavlink_router `udp:14540`, `conn/timesync_mode:=NONE` (штампует IMU sim-временем, иначе «imu message in disorder»). Через 20 с поднимает stream_rate IMU до 200 Гц (по умолчанию ArduPilot шлёт ~7 Гц, VINS нужно ≥100).

## Важные моменты

- **Идемпотентен**: каждый шаг проверяет `pgrep`/наличие файла перед запуском, поэтому повторный прогон (`make nav`) не плодит дубли нод и не пересобирает зря.
- **Почему байеризатор и ноды по отдельности** (`nohup`, не один launch): если байеризатор падает внутри launch, launch-менеджер убивает всё дерево. Здесь он изолирован, а `nav_up.sh` гарантирует порядок: сначала байеризатор + ожидание `/dev/rawbayer`, потом launch.
- Логи всех нод — в `docker/sim/output/` (`bayerizer.log`, `sim_nav.log`, `mavros.log`); читаются с хоста (`make logs`).
- Парный скрипт на стороне Gazebo — `sim_up.sh` (поднимает Gazebo+SITL+bridge в контейнере `simulator`).


--------------------------------
Что делает `sim_up.sh` и как он запускается?
--------------------------------

`sim_up.sh` (`docker/sim/scripts/sim_up.sh`) — это **entrypoint стороны симулятора**: один скрипт, который внутри контейнера `simulator` поднимает Gazebo, ArduPilot SITL и мост Gazebo→ROS2. Парный к `nav_up.sh` (тот — nav-сторона: камера/VINS/MAVROS). Пишет в лог финальное «simulator: готово».

## Как запускается

- **Автоматически** — прописан как `command:` контейнера `simulator` в `docker-compose.yml`, стартует сам при `up`/`start`/`restart-all`.
- **Вручную** — `make sim` (= `docker exec -i p1317_simulator bash -s < scripts/sim_up.sh`). Смонтирован как `/scripts/sim_up.sh:ro` — правки на хосте применяются без пересборки образа.

Запускается через `bash -s` (неинтерактивный шелл): ENV из Dockerfile есть (PATH к SITL), но `.bashrc` не читается, поэтому ROS подключается явно (`source /opt/ros/humble/setup.bash`).

## Что он делает по шагам

1. **Патч разрешения камеры** (если `CAMERA_W`/`CAMERA_H` ≠ 1280×720): SDF статичен (gz не подставляет env), поэтому кладёт ПАТЧЕНУЮ копию модели `iris_cam` в `/tmp/sim_models`, правит `<width>/<height>` через `sed` и выводит её первой в `GZ_SIM_RESOURCE_PATH` — репозиторную модель не трогает (git чист).
2. **Gazebo Harmonic** (`nohup`) — `gz sim -s --headless-rendering` мир `WORLD` (default `mili_fortress.sdf`); `ArduPilotPlugin` слушает порт 9002.
3. **`sleep 5`** — дать Gazebo поднять физику/плагины до подключения SITL.
4. **ArduPilot SITL** (`nohup`) — `sim_vehicle.py -v ArduCopter -f gazebo-iris --model JSON`, подключается к плагину@9002, MAVLink на tcp:5760. `--no-mavproxy` (телеметрию разводит mavlink_router), `--no-rebuild`. Применяет два .parm: `gazebo-iris.parm` + `sitl-extra.parm`.
5. **`ros_gz_bridge`** (`nohup`) — мост Gazebo→ROS2 ровно двух топиков: `/camera/image_raw` и **`/clock`** (источник sim-времени для всех nav-нод).

## Важные моменты

- **Идемпотентен**: каждый шаг проверяет `pgrep` (`gz sim`/`sim_vehicle`/`ros_gz_bridge`) перед запуском — повторный `make sim` не плодит дубли.
- **`ros_gz_bridge` здесь, а не в nav** и БЕЗ `use_sim_time` — он ИСТОЧНИК `/clock`, ему sim-время ставить нельзя. В мост заведены только камера и часы; **IMU через bridge НЕ идёт** (он приходит в ROS через SITL→MAVROS, см. `src/sim/CLAUDE.md`).
- **SITL-параметры эфемерны**: критичные (`SCHED_LOOP_RATE 100`, `ARMING_CHECK 0` и др.) применяются в `.parm` внутри контейнера и теряются на `fresh-start` (пересоздание контейнера); `sitl-extra.parm` смонтирован с хоста и переживает. См. CLAUDE.md п.12.
- Логи — в `docker/sim/output/` (`gz_sim.log`, `sitl.log`, `ros_gz_bridge.log`); читаются с хоста (`make logs`).
- **`depends_on`**: `nav` ждёт `simulator` — `nav_up.sh` ждёт активации `/dev/rawbayer`, который зависит от потока камеры из Gazebo.


--------------------------------
Что такое `docker-compose.cpu.yml` и когда он нужен?
--------------------------------

`docker-compose.cpu.yml` — это **CPU-оверрайд** симуляционного стека: тонкий compose-файл, который кладётся ПОВЕРХ базового `docker-compose.yml` и переключает стек в режим работы **без GPU** (ветка `nn2_c3_cpu`, пока T4 на GCE в дефиците). Базовый compose он не меняет — переопределяет только нужные поля.

## Когда нужен

Когда прогоняешь симуляцию на машине **без NVIDIA-драйвера** (CPU-бокс). Боевой Orin и штатный GPU-sim остаются на базовом compose (`camera_node` + `runtime: nvidia`) — этот файл их не трогает.

## Что он переопределяет

| Поле | База | CPU-оверрайд | Зачем |
|---|---|---|---|
| `runtime` (обоих сервисов) | `nvidia` | `runc` | без NVIDIA-драйвера nvidia-runtime не стартует; `NVIDIA_*`-env при этом безвредны |
| `simulator` GL | EGL/NVIDIA | `LIBGL_ALWAYS_SOFTWARE=1`, `GALLIUM_DRIVER=llvmpipe` | софтовый рендер камеры Gazebo вместо GPU |
| `nav` камера | `camera_node` (CUDA) | `CAMERA_NODE=camera_node_cpu` | `sim_nav.launch.py` поднимает drop-in CPU-дебайер (`cv::*` вместо `cv::cuda::*`) |
| разрешение (оба) | 1280×720 | `CAMERA_W=320`, `CAMERA_H=180` | llvmpipe на 1280×720 тянет ~2 Гц; 320×180 (×16 меньше пикселей) → ~30 Гц |

`CAMERA_W/H` ставится **в обоих** сервисах одинаково — `sim_up.sh` патчит SDF Gazebo-камеры, `bayerizer`/`camera_node` берут тот же размер, интринсики VINS пересчитываются из `sim.yaml`.

## Как применять

Накладывается двумя `-f` (порядок важен — оверрайд вторым):

```bash
docker compose -f docker-compose.yml -f docker-compose.cpu.yml up -d
```

Через Makefile — просто **`CPU=1`** к любому таргету (он подставляет оба `-f`):

```bash
make CPU=1 build && make CPU=1 up && make CPU=1 logs
make CPU=1 restart-all && make CPU=1 wait
```

⚠️ Флаг держать на **каждой** команде прогона, иначе часть пойдёт по базовому compose (с `runtime: nvidia`) и не совпадёт.

## Замечания

- Разрешение 320×180 здесь — **не** ради ускорения рендера (на RTF разрешение почти не влияет, горло — физика+геометрия сцены `mili_fortress`), а ради экономии CPU на nav-стороне и того, что llvmpipe масштабируется по числу пикселей под заполнение. Подробно про RTF и софтрендер — `src/sim/CLAUDE.md`.
- `camera_node_cpu` — drop-in замена боевой ноды (тот же код через `camera_core.hpp`, CPU-дебайер вместо CUDA); см. `src/camera/`.


--------------------------------
Что такое `camera_node_cpu` и чем он отличается от боевого `camera_node`?
--------------------------------

`camera_node_cpu` (`src/camera/camera_node_cpu.cpp`) — это **drop-in CPU-версия** камера-ноды: тот же V4L2-захват и те же топики, что у боевого `camera_node`, но дебайер и gain считаются на CPU (`cv::*`) вместо GPU (`cv::cuda::*`). Нужен для прогона на машинах **без NVIDIA GPU/драйвера** (GPU-less sim, ветка `nn2_c3_cpu`). Боевой Orin и штатный GPU-sim остаются на `camera_node` — этот executable их не трогает.

## Общая база (почему «drop-in»)

Обе ноды наследуют `V4L2CameraBase` (`camera_core.hpp`), который держит всё общее: V4L2-захват, чтение кадра как `CV_16UC1`, нормализацию 16→8 (`NORM_MINMAX`), гамму, публикацию `/image_mono` (вход VINS) + `/image_color` (nav) + `/camera_info`, штамп `get_clock()->now()` (уважает `use_sim_time`). Различается **только** один виртуальный метод `process_frame()` — собственно дебайер + gain. Поэтому подмена ноды не меняет ни топики, ни тайминг, ни параметры.

## В чём именно разница

| | `camera_node` (боевой) | `camera_node_cpu` |
|---|---|---|
| Дебайер | `cv::cuda::demosaicing` (GPU) | `cv::cvtColor` (CPU) |
| Код Байера | `BayerGB2RGB` | `BayerGR2BGR` |
| Цвет→BGR | отдельный `cv::cuda::cvtColor RGB→BGR` | `cvtColor` сразу даёт BGR |
| gain | на GPU | `cv::split`/`convertTo`/`merge` по BGR-каналам |
| OpenCV-линковка | полный (вкл. CUDA-модули) | только `opencv_core/imgproc/videoio` — **без CUDA-рантайма** |

⚠️ Разные коды Байера (`BayerGB2RGB` в CUDA vs `BayerGR2BGR` на CPU) — это **норма**, а не баг: в OpenCV CUDA-модуле сдвиг в именовании паттернов, оба обрабатывают один физический паттерн GRBG сенсора (см. CLAUDE.md, раздел про камеру). На CPU `cvtColor` отдаёт сразу BGR, поэтому отдельный RGB→BGR swap не нужен, а gain раскладывается прямо по BGR-каналам (ch0=B, ch1=G, ch2=R).

## Как он выбирается

Не правя код/launch — через env **`CAMERA_NODE`**, который читает `sim_nav.launch.py`:
- по умолчанию `camera_node` (CUDA);
- CPU-оверрайд compose (`docker-compose.cpu.yml`) ставит `CAMERA_NODE=camera_node_cpu`.

То есть на практике он включается вместе со всем CPU-режимом — `make CPU=1 …` (см. запись про `docker-compose.cpu.yml`).

## Замечания

- Это отдельный executable (`add_executable(camera_node_cpu …)` в `CMakeLists.txt`), линкуемый **без** CUDA-модулей — бинарь не тянет зависимостей на GPU-рантайм, поэтому стартует на машине без драйвера.
- Поскольку база общая, `/image_mono` для VINS считается ровно так же (mono из корректного BGR через `BGR2GRAY`) — поведение VINS от подмены ноды не меняется, отличается только устройство дебайера.


--------------------------------
Что такое `sim.yaml` и чем он отличается от боевого конфига?
--------------------------------

`sim.yaml` (`src/vins/VINS-MONO-ROS2/config_pkg/config/sim.yaml`) — это **конфиг VINS-Mono для симуляции** (Gazebo + SITL). Аналог боевого `dummy_13_7.yaml` (реальный ArduCam), но подогнан под идеальную Gazebo-камеру и sim-окружение. Его подхватывают `feature_tracker` и `vins_estimator` (через `config_file` в `sim_nav.launch.py`).

## Чем отличается от боевого `dummy_13_7.yaml`

| | Боевой (ArduCam) | `sim.yaml` (Gazebo) |
|---|---|---|
| Дисторсия | реальная (линза) | **нулевая** — Gazebo-камера идеальный pinhole |
| Интринсики | калибровка камеры | из `model.sdf`: fov 90° @ 1280×720 → `fx=fy=640`, `cx=640`, `cy=360` |
| Пути | `/root/vins_ws/` | `/root/sim_ws/` |
| Экстринсики камера↔IMU | с реальной рамы | из позы камеры в SDF (поза `0.15 0 0.05`, pitch 0.26) |
| Шум IMU (`acc_n`/`gyr_n`…) | поднят под вибрации рамы | **занижен** — в sim вибраций нет |
| `td` (камера↔IMU) | оценивается | `estimate_td: 0`, `td: 0.0` |

## Почему так

- **Нулевая дисторсия + простые интринсики** — Gazebo рендерит идеальный pinhole, считать искажения не нужно; `fx = (width/2)/tan(fov/2) = 640/tan(45°) = 640`.
- **`td=0`, оценку выкл** — камера и IMU оба штампуются sim-clock'ом (`use_sim_time`), так что смещение между ними нулевое (см. `src/sim/CLAUDE.md`, раздел про два времени). Если инициализация VINS будет страдать на штампах — рычаг тут: включить `estimate_td: 1`.
- **Заниженный шум IMU** — в симуляции нет вибраций рамы, поэтому `acc_n/gyr_n` возвращены к базовым (в боевом они подняты ~×10). Комментарий в файле: «check2» был из-за ill-conditioned ковариации, не из-за `acc_n`.
- **Экстринсики** считаются из позы камеры в SDF: `extrinsicRotation` (RIC, тело FLU → camera-optical) выведена из наклона 0.26 рад, `extrinsicTranslation = [0.15, 0, 0.05]`. `estimate_extrinsic: 0` — фиксируем, не оптимизируем.

## ⚠️ Разрешение должно быть согласовано

Разрешение в `sim.yaml` (1280×720) обязано совпадать с камерой Gazebo (`model.sdf`) и `width/height` в camera_node — иначе интринсики не соответствуют картинке и VINS «поедет».

При CPU-режиме (320×180) **отдельный .yaml не правится вручную**: `sim_nav.launch.py::_vins_config()` масштабирует `image_width/height` и `fx/fy/cx/cy` из этого же `sim.yaml` в `/tmp/sim_WxH.yaml` (один источник, чтобы не дрейфовал). На 1280×720 возвращается `sim.yaml` как есть. См. запись про `sim_nav.launch.py`.

## Замечания

- Топики, прописанные тут: `imu_topic: /mavros/imu/data_raw`, `image_topic: /image_mono` (вход VINS — это mono-выход camera_node).
- `freq: 10` — feature_tracker дросселирует публикацию фич до 10 Гц (sim-Гц), отвязывая нагрузку estimator от сырого fps камеры.
- Это конфиг под Gazebo-мир; реальная камера/полётник — `dummy_13_7.yaml`.


--------------------------------
Что такое `dummy_13_7.yaml` и чем он отличается от `sim.yaml`?
--------------------------------

`dummy_13_7.yaml` (`src/vins/VINS-MONO-ROS2/config_pkg/config/dummy_13_7.yaml`) — это **боевой конфиг VINS-Mono** для реального борта (ArduCam на Jetson Orin). Его подхватывают `feature_tracker`/`vins_estimator` на Orin. Симуляционный аналог — `sim.yaml`.

«dummy» в имени — потому что это **стартовый/некалиброванный** конфиг: интринсики и шум проставлены ориентировочно, а экстринсики и временную задержку VINS должен **довести сам онлайн** (см. ниже). Это рабочая заготовка под реальную камеру, а не финальная калибровка.

## Чем отличается от `sim.yaml`

| | `dummy_13_7.yaml` (боевой) | `sim.yaml` (Gazebo) |
|---|---|---|
| `estimate_td` | **1** — VINS сам оценивает задержку камера↔IMU | 0 (оба на sim-clock → td=0) |
| `estimate_extrinsic` | **1** — оптимизирует экстринсики от начального guess | 0 — зафиксированы из SDF |
| `extrinsicRotation` | единичная матрица (как стартовый guess) | выведена из позы камеры в SDF |
| `extrinsicTranslation` | `[0.14, 0, -0.06]` (реальная установка) | `[0.15, 0, 0.05]` (из SDF) |
| `freq` (трекер фич) | 15 | 10 |
| Пути | `/root/vins_ws/` | `/root/sim_ws/` |
| Шум IMU | поднят под вибрации рамы | занижен (в sim вибраций нет) |

## Почему так (боевая специфика)

- **`estimate_td: 1`** — на реальном железе камера и IMU идут разными путями с аппаратной задержкой, которая заранее неизвестна → пусть VINS подстраивает её онлайн. В симуляции оба на sim-clock, поэтому там `td=0` фиксирован.
- **`estimate_extrinsic: 1` + единичная RIC** — точное крепление камера↔IMU на борту не измерено идеально; даём грубый начальный guess и позволяем VINS откалибровать вокруг него (в этом форке: 0=fix, 1=optimize, 2=no prior/calibrate). В sim геометрия известна точно из SDF → фиксируем.
- **Поднятый шум IMU** — реальная рама вибрирует, поэтому `acc_n`/`gyr_n` увеличены относительно базовых (компенсация вибраций); в `sim.yaml` они возвращены к базовым.
- Интринсики/дисторсия совпадают с `sim.yaml` по числам (PINHOLE, `fx=fy=640`, `cx=640/cy=360`, нулевая дисторсия) — это заготовка под 1280×720; под реальную камеру их положено уточнить калибровкой.

## Замечания

- Топики те же, что в sim: `imu_topic: /mavros/imu/data_raw`, `image_topic: /image_mono`.
- Это конфиг для боевого стека `docker/orin/`; в симуляции используется `sim.yaml` (его выбирает `sim_nav.launch.py`).


--------------------------------
Что такое `mavlink_router` и зачем он в стеке?
--------------------------------

`mavlink_router` — это третий контейнер симуляционного стека (рядом с `simulator` и `nav`): **единая точка раздачи MAVLink-телеметрии**. ArduPilot SITL поднимает ОДИН TCP-сервер на `127.0.0.1:5760`, а потребителей несколько (MAVROS, QGroundControl, кастомные узлы). Чтобы они не дрались за единственный порт, `mavlink-router` подключается к SITL клиентом и веером раздаёт поток по UDP.

```
SITL (tcp:5760) → mavlink-router ─┬─► UDP 14550  QGroundControl
                                  ├─► UDP 14540  MAVROS (контейнер nav)
                                  └─► UDP 14541  кастомный C++/Python узел
```

## Зачем он нужен

TCP-сервер SITL принимает соединения, но «размножать» один поток на несколько независимых клиентов с разными нуждами неудобно и хрупко. `mavlink-router` — стандартное решение: один аплинк к FCU, много даунлинков. В concept.txt был конфликт (два сервера на 5760) — он этим и снят: **router подключается клиентом** к tcp:5760, а наружу отдаёт UDP-эндпоинты.

## Конфиг

`docker/sim/config/mavlink-router.conf`, смонтирован в контейнер как `/etc/mavlink-router/main.conf:ro` (правки на хосте — без пересборки образа):
- `[TcpEndpoint sitl]` `Mode=Client`, `127.0.0.1:5760` — источник (SITL).
- `[UdpEndpoint qgc]` → 14550, `[UdpEndpoint mavros]` → 14540, `[UdpEndpoint cpp_node]` → 14541 — потребители.
- `TcpServerPort=0` (свой TCP-сервер не поднимает), `MavlinkDialect=ardupilotmega`.

MAVROS в `nav_up.sh` цепляется именно к 14540 (`fcu_url:=udp://:14540@127.0.0.1`).

## Как устроен контейнер

- Отдельный образ (`docker/sim/mavlink_router/Dockerfile`): ubuntu:22.04, собирает `mavlink-router` из исходников (meson/ninja). `CMD mavlink-routerd -c /etc/mavlink-router/main.conf`.
- `network_mode: host` — чтобы UDP-эндпоинты на `127.0.0.1` были видны MAVROS и QGC на хосте.
- `depends_on: simulator` — источник (SITL) живёт в контейнере `simulator`.
- `restart: unless-stopped`.

## Замечания

- Это единственный контейнер стека **без** ROS и без GPU — чистый MAVLink-маршрутизатор.
- Если MAVROS не видит FCU (`make status` не показывает `FCU: ArduCopter`) — проверять цепочку: SITL поднял tcp:5760? router подключился клиентом? MAVROS слушает 14540? Порты — в таблице «Порты MAVLink» в `README.md`.
- QGroundControl (14550) можно подключить с хоста для ручного наблюдения за дроном в прогоне.
