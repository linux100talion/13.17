FAQ_gps.txt — почему арм в симуляции долгий и как обойти GPS
============================================================

ВОПРОС
------
Арм происходит долго. Можно ли его ускорить? В частности — обойти GPS, чтобы
арм не ждал позиционную оценку?


КОНТЕКСТ / ДИАГНОЗ
-----------------
Арм блокируется сообщением FCU "Arm: Need Position Estimate" и снимается только
событием "EKF3 IMU0 is using GPS". Таймлайн старта FCU (run3, wall-время, RTF≈0.07):

    EKF3 IMU0 initialised         +0с
    tilt alignment complete       +18с
    origin set                    +216с   (~14 sim-сек: EKF ждёт первый фикс GPS)
    is using GPS                  +510с   (~20 sim-сек: EKF принимает GPS) → арм

Итого от инициализации EKF до "is using GPS" ≈ 35 sim-сек (≈500 wall-сек на
RTF≈0.07). Это доминирующая часть ожидания арма.

Важно: гейт — НЕ высота спавна дрона (это отдельно проверено и исправлено,
спавн теперь ровно на землю z=0.245, дрон статичен — на время арма не влияет).
Гейт — момент появления/готовности фикса GPS в EKF.

ЧТО НЕ ПОМОГЛО:
  EK3_GPS_CHECK 0 (+ GPS_DELAY_MS 0) — без эффекта (510 vs 502 wall-сек).
  Значит дело НЕ в предпроверках качества GPS в EKF, а в самом времени появления
  фикса GPS + сходимости EKF (~35 sim-сек). Это фундаментально для ArduCopter SITL
  и плохо тюнится параметрами. Реальный множитель боли — RTF≈0.07 на CPU-боксе.


ПОДВОДНЫЙ КАМЕНЬ ОБХОДА GPS
--------------------------
Просто «не ждать GPS» для арма недостаточно: GUIDED-взлёт и квадрат ТОЖЕ требуют
позиционную оценку. Если убрать GPS и не дать замену — взлёт/квадрат не поедут.
А «настоящая» замена (VINS) на старте недоступна: VINS инициализируется только
в полёте (нужен параллакс + IMU excitation) — классическая курица-яйцо, ради
чего GPS и стоит костылём в симуляции.

Состояние обвязки (разведка):
  - В дефолтном gazebo-iris.parm EK3_SRC* не заданы → EKF берёт GPS (POSXY=3).
  - ray_tracer умеет писать /mavros/vision_pose/pose (External Nav), но ТОЛЬКО
    когда есть скорректированная одометрия от VINS — на старте её нет.
  - Боевой план проекта и так: EK3_SRC1_POSXY=6 + позиция через vision_pose.


ЧИСТОЕ РЕШЕНИЕ ДЛЯ СИМУЛЯЦИИ (предложено, НЕ реализовано)
--------------------------------------------------------
Скармливать EKF ground-truth позу из Gazebo как vision_pose (sim-чит):

    gz pose дрона (точная) → republisher → /mavros/vision_pose/pose (20 Гц)
                                           + EK3_SRC1_POSXY=6, GPS_TYPE=0

Тогда: EKF получает корректную позицию с t=0 → арм БЕЗ ожидания лока GPS,
и позиция верна ВЕСЬ полёт (квадрат работает). GPS выключаем полностью.
Совпадает с боевой архитектурой (тот же vision_pose-вход), только источник
в sim — Gazebo-правда вместо VINS/ray_tracer.

Что нужно сделать (3 части, ~1 тест-цикл):
  1. sitl-extra.parm: EK3_SRC1_POSXY 6, EK3_SRC1_VELXY 6, EK3_SRC1_POSZ 1 (баро),
     GPS_TYPE 0.
  2. Бридж ground-truth позы дрона из Gazebo в ROS (добавить в ros_gz_bridge).
  3. Мелкий republisher → /mavros/vision_pose/pose (в nav_up.sh, как запрос
     IMU-стрима).

Риск: перенастройка источника EKF — может потребовать пары итераций по
yaw/origin (~8 мин каждая на CPU-боксе). Ожидаемый результат: арм за единицы
sim-секунд вместо ~35.


АЛЬТЕРНАТИВЫ (отвергнуты/частичные)
----------------------------------
  - Дожать GPS/EKF параметрами — низкая уверенность (GPS_CHECK уже не дал эффекта).
  - Поднять RTF (снять нагрузку CPU в предармовой фазе: отложить камеру/VINS/
    байеризатор до арма) — самая большая отдача по wall-времени, но перестройка
    стартовой последовательности.
  - Арм в non-GPS режиме (ALT_HOLD/STABILIZE) — ускоряет сам факт арма, но
    время до ПОЛЁТА не сокращает (GUIDED-взлёт всё равно ждёт позицию).
  - Поднять ARM_SIM_BUDGET в arm.sh — убирает ложное ⚠️ в arm.sh, но время не режет.
