================================================================================
FAQ — capture_scene.sh (атомарный прогон диагностики камеры с хоста)
================================================================================

Скрипт: src/lab/capture_scene.sh (+ src/lab/extract_frames.py).
Запускается С ХОСТА, поднимает весь стек и проходит прогон одной командой.


--------------------------------
Что делает `capture_scene.sh`?
--------------------------------

Единый АТОМАРНЫЙ прогон «от рестарта стека до результата» — СЕКВЕНСОР команд: сам
поднимает стек, проигрывает заданную ПОСЛЕДОВАТЕЛЬНОСТЬ лётных команд с записью
rosbag камеры, вытаскивает кадры по пройденному пути и заливает их на Google
Drive. Это обёртка над make-целями и скриптами src/lab — эталон «дисциплины
прогона» (не ручные команды в контейнере, а один скрипт). Назначение —
диагностика того, ЧТО реально видит камера дрона (для отладки инициализации VINS).

Синтаксис:

    capture_scene.sh [WxH] <команда> [арг] <команда> [арг] ...

    cd docker/sim && make capture-scene        # дефолтная последовательность (CSARGS)
    make capture-scene CSARGS="640x480 arm takeoff 5 hover 2 land"
    # или напрямую С ХОСТА из любого места:
    bash src/lab/capture_scene.sh 640x480 arm takeoff 5 hover 2 land


--------------------------------
Какие есть команды?
--------------------------------

Первый позиционный аргумент — ОПЦИОНАЛЬНОЕ разрешение WxH (напр. 640x480). Если
задано → стек ПЕРЕСОЗДАЁТСЯ (fresh-start), т.к. env применяется при создании
контейнера; иначе быстрый restart-all. Далее — последовательность из 5 чистых
лётных команд (каждая = свой скрипт в /lab/, исполняется по порядку):

    arm             GUIDED + арм (без взлёта)              /lab/arm.sh
    takeoff [ALT]   взлёт на ALT м (default 3)             /lab/takeoff.sh
    hover [SIM_SEC] висение SIM_SEC сек SIM-времени (def 10) /lab/hover.sh
    land            посадка (режим LAND)                   /lab/land.sh
    disarm          дизарм                                 /lab/disarm.sh

takeoff/hover съедают следующий числовой токен как аргумент; arm/land/disarm —
без аргумента. Неизвестная команда → ошибка (до рестарта стека, впустую не
поднимаем). Команды ждут ПО ФАКТУ (поллинг mode/armed/z; hover — по /clock),
а не фиксированными sleep — RTF-независимо.

hover = SIM-секунды (не wall): в GUIDED коптер сам удерживает точку после
takeoff, поэтому hover просто ждёт прироста sim-времени по /clock. При низком
RTF (≈0.05) фикс. wall-секунды были бы мизером sim-времени.


--------------------------------
Какие есть параметры (env)?
--------------------------------

Полётные параметры теперь ПОЗИЦИОННЫЕ (команды), а env управляет только
рестартом / записью / извлечением / заливкой:

| Env                      | Default                  | Что |
|--------------------------|--------------------------|-----|
| RESTART                  | 1                        | 1 = перезапуск стека (restart-all/fresh-start + wait); 0 = на живом стеке (⚠️ рассинхрон) |
| RECORD                   | 1                        | 1 = писать rosbag (/image_color + поза) вокруг всей последовательности |
| DIST_M                   | 0.5                      | шаг выборки кадров по пройденному пути, м |
| N_FRAMES                 | 30                       | макс. число кадров (0 = без лимита) |
| TOPIC                    | /image_color             | топик камеры |
| POSE_TOPIC               | /mavros/local_position/pose | поза для расчёта пути |
| CPU                      | —                        | CPU=1 → GPU-less режим (накладывает docker-compose.cpu.yml) |
| GDRIVE_UP                | 1                        | 1 = заливать на Drive; 0 = только снять кадры |
| GDRIVE_REMOTE / GDRIVE_DIR | gdrive / 13.17/scene_img | rclone-remote и папка на Drive |

Разрешение задаётся ПОЗИЦИОННО (WxH), не через env. При RECORD=1 старые rosbag'ы
(output/scene_bag*) удаляются в начале; свежий bag этого прогона (2+ ГБ)
ОСТАЁТСЯ в docker/sim/output/scene_bag для анализа.

Примеры:

    bash src/lab/capture_scene.sh 640x480 arm takeoff 5 hover 2 land    # 640×480 (fresh-start)
    bash src/lab/capture_scene.sh arm takeoff 3 hover 20 land disarm    # без смены разрешения (restart-all)
    DIST_M=1.0 bash src/lab/capture_scene.sh arm takeoff 4 hover 20 land # выборка реже — каждый метр пути
    GDRIVE_UP=0 bash src/lab/capture_scene.sh arm takeoff 4 hover 20 land # снять кадры локально, без заливки
    RECORD=0 bash src/lab/capture_scene.sh arm takeoff 3 land            # дешёвая проверка взлёта (без bag)


--------------------------------
Как выбираются кадры — почему «по пути», а не по времени?
--------------------------------

Кадры выбираются по ПРОЙДЕННОМУ ПУТИ дрона, а не по таймеру. Это удобнее для
анализа сцены: кадры равномерны в ПРОСТРАНСТВЕ (а не во времени), число кадров
пропорционально длине маршрута.

Как работает (extract_frames.py, внутри nav-контейнера, rosbag2_py + cv_bridge):
- запись пишет в bag ДВА топика — /image_color И /mavros/local_position/pose;
- скрипт копит 3D-длину пути между позами и сохраняет: первый кадр (старт) +
  каждый раз, как с прошлого сохранения набежало >= DIST_M метров;
- имя файла несёт пройденный путь: frame_03_001.50m.jpg.

ВАЖНО: если дрон не двигался (не взлетел) или позы в bag нет → останется ТОЛЬКО
первый кадр + предупреждение. На время НЕ откатываемся.

Извлечение можно перезапустить отдельно по уже снятому bag (с другим шагом), не
делая новый прогон — внутри nav-контейнера:

    docker exec -e SCENE_DIST_M=0.5 -e SCENE_N=0 p1317_nav bash -lc \
      'source /opt/ros/humble/setup.bash; source /opt/overlay/install/setup.bash; \
       source /root/sim_ws/install/setup.bash; python3 /lab/extract_frames.py'

Env extract_frames.py: SCENE_BAG (default …/output/scene_bag), SCENE_OUT
(…/output/scene_img), SCENE_TOPIC (/image_color), SCENE_POSE
(/mavros/local_position/pose), SCENE_DIST_M (0.5), SCENE_N (30; 0 = без лимита).
Требует overlay /opt/overlay (cv_bridge против CUDA-OpenCV).


--------------------------------
Как задать разрешение камеры для прогона?
--------------------------------

Разрешение задаётся ПЕРВЫМ позиционным аргументом WxH (напр. 640x480).
capture_scene.sh парсит его в CAMERA_W/CAMERA_H, которые растекаются по всем 5
точкам автоматически (SDF-камера Gazebo, bayerizer, camera_node, интринсики VINS,
CameraInfo). docker-compose.yml интерполирует их из env хоста (${CAMERA_W:-1280} /
:-720; CPU-оверрайд → :-320 / :-180).

ПОДВОХ: env применяется при СОЗДАНИИ контейнера — restart-all (stop/start) его НЕ
перечитывает. Поэтому при заданном WxH `capture_scene.sh` в фазе рестарта делает
fresh-start (пересоздание контейнеров), а не restart-all. Это безопасно:
критичные SITL-параметры (SCHED_LOOP_RATE 100, ARMING_CHECK 0, DISARM_DELAY 0,
FS_GCS_ENABLE 0) лежат в host-смонтированном config/sitl-extra.parm и
применяются sim_up.sh при каждом старте.

    bash src/lab/capture_scene.sh 640x360 arm takeoff 4 hover 20 land       # 640×360 на GPU
    CPU=1 bash src/lab/capture_scene.sh 320x180 arm takeoff 4 hover 20 land # CPU-бокс

⚠️ При RESTART=0 разрешение НЕ применится (нет пересоздания) — скрипт предупредит.


--------------------------------
Сколько идёт прогон по времени?
--------------------------------

~4–7 мин на типовой прогон. Складывается из:
  restart-all (~10–20с; fresh-start при смене разрешения чуть дольше) + make wait
  до «nav: готово» (~60–120с) + прогрев EKF (~85с) + arm/takeoff (10–40с при
  успехе, до 180с при отказе) + hover (зависит от SIM_SEC и RTF) + извлечение
  кадров (~15с) + заливка на Drive (~18с).
Разброс даёт взлёт: при зависшем takeoff ретрай до таймаута → +3 мин. Дешёвая
проверка взлёта (короткая последовательность без hover, RECORD=0) — заметно быстрее.


--------------------------------
Что нужно для заливки на Google Drive?
--------------------------------

Заливка через rclone (remote по умолчанию gdrive:). Если remote не найден —
кадры остаются локально в docker/sim/output/scene_img, скрипт выходит с ошибкой
на стадии заливки. Бокс headless → OAuth проходят в Google Cloud Shell и копируют
готовый rclone.conf на бокс. Детали — src/lab/gdrive_rclone_setup.md /
src/lab/CLAUDE.md. GDRIVE_UP=0 — пропустить заливку (снять кадры локально).
